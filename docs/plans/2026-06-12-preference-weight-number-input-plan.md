# 매칭 가중치 숫자 입력 UI 전환 계획

## Summary

현재 `/profile/preferences`는 가중치를 슬라이더로 입력하고, 한 항목을 움직이면 나머지 항목이 자동으로 재분배된다. 이 방식은 합계 100%를 쉽게 맞추는 장점은 있지만, 사용자가 "내가 방금 무엇을 얼마나 바꿨는지"를 이해하기 어렵다.

수정 방향은 **슬라이더 제거 + 숫자 직접 입력 + 가중치 설명 강화**다. 사용자가 각 항목의 의미를 먼저 이해하고, 직접 퍼센트를 입력해서 총합 100%를 맞추게 만든다.

## Current Problem

- 한 항목을 드래그하면 다른 항목들이 자동으로 바뀐다.
- 사용자는 외모를 20%에서 35%로 올렸을 때 성격/학교/취미가 왜 줄어드는지 즉시 이해하기 어렵다.
- 슬라이더는 모바일에서 미세 조정이 어렵고, 숫자 의도가 명확하지 않다.
- "자동으로 100%가 맞춰져"라는 문구는 편해 보이지만 실제로는 사용자의 선택권을 흐린다.
- 가중치 항목이 매칭 점수에 어떤 식으로 반영되는지 설명이 부족하다.

## Target UX

### 1. 상단 설명

제목:

```text
매칭에서 무엇을 더 중요하게 볼까?
```

보조 문구:

```text
각 항목에 0-100 사이 숫자를 입력해줘. 합계가 정확히 100%가 되어야 저장할 수 있어.
```

상단 고정 요약:

```text
현재 합계 86% · 남은 14%
```

합계가 100%면:

```text
합계 100% · 저장 가능
```

합계가 100%를 넘으면:

```text
합계 112% · 12%를 줄여야 해
```

### 1-1. 이미지 기반 설명 레이어

숫자 입력만 있으면 사용자가 "내가 뭘 원하는지"보다 "합계를 맞추는 일"에 집중하게 된다. 따라서 화면 상단 또는 각 항목 카드 안에 **가중치가 실제 매칭에서 어떤 의미인지 보여주는 이미지/시각 장면**을 넣는다.

핵심 메시지:

```text
이 비율은 상대를 평가하는 점수가 아니라,
이번 매칭에서 내가 더 중요하게 보고 싶은 기준이야.
```

이미지 방향:

- 생성 이미지는 **글자 없는 추상 이미지**로 만든다.
- 한글 문구는 이미지 안에 넣지 않고, 앱의 HTML 텍스트로 렌더링한다.
- 사진처럼 특정 사람 얼굴이 보이는 이미지는 피한다. 프로필 비공개 원칙과 충돌할 수 있다.
- "상대 점수화"처럼 보이지 않도록 게이지/등급/랭킹 느낌은 약하게 쓴다.
- 대신 "내가 원하는 만남의 장면", "중요도 조합", "매칭 레시피"처럼 보이게 한다.

추천 비주얼 콘셉트:

```text
Matching Recipe
```

사용자가 100%라는 한 그릇을 직접 배합하는 느낌이다. 외모, 성격, 시간대, 취미 등이 각각 다른 색의 재료처럼 보이고, 사용자가 숫자를 바꾸면 내 매칭 레시피가 달라진다는 인상을 준다.

대체 콘셉트:

```text
Matching Compass
```

사용자의 우선순위가 나침반 방향처럼 표시된다. 외모를 높이면 외모 방향으로, 성격을 높이면 성격 방향으로 중심이 이동한다. "내가 어떤 만남을 찾는지"를 직관적으로 보여주기 좋다.

1차 구현 추천은 `Matching Recipe`다. 숫자 입력과 합계 100% 개념이 자연스럽게 연결되고, 설명도 쉽다.

### 2. 입력 카드 구조

각 항목은 카드 하나로 구성한다.

```text
외모
상대의 외모 취향/사진 분석 결과가 내 선호와 얼마나 맞는지 보는 비중

[ 25 ] %
```

숫자 입력 옆에는 `-5`, `+5` 스텝 버튼을 둘 수 있다. 모바일에서 키보드 입력이 불편한 사용자를 위한 보조 수단이다.

카드 안에는 작은 추상 비주얼 또는 아이콘형 일러스트를 둔다.

| 항목 | 이미지 느낌 |
|---|---|
| 외모 | 빛나는 실루엣, 거울, 부드러운 하이라이트 |
| 성격 | 겹쳐지는 말풍선, 따뜻한 대화 흐름 |
| 키 | 높낮이가 다른 두 선, 스케일/자 |
| 체형 | 균형 잡힌 형태, 자세/핏의 추상 실루엣 |
| 학교 | 캠퍼스 아치, 책, 학생증 느낌 |
| 취미 | 작은 오브젝트들이 모이는 콜라주 |
| 시간대 | 겹치는 캘린더 링, 시계 궤도 |

이 이미지는 "해당 항목이 무엇인지"를 설명하는 보조 장치다. 실제 매칭 계산값이나 상대방 정보를 보여주는 것이 아니다.

### 3. 자동 보정 제거

핵심 원칙:

- 사용자가 입력한 항목만 바뀐다.
- 다른 항목은 자동으로 바꾸지 않는다.
- 합계가 100%가 아니면 저장 버튼을 비활성화한다.
- 저장 버튼 아래에 부족/초과 안내를 보여준다.

이렇게 하면 사용자는 "내가 직접 정한 비율"이라는 감각을 가진다.

## Weight Meanings

| 항목 | 사용자 설명 | 내부 의미 |
|---|---|---|
| 외모 | 상대 외모가 내 취향과 맞는지 | 이상형 월드컵/사진 분석 기반 외모 선호 적합도 |
| 성격 | 대화 스타일과 성향이 맞는지 | 성격 선호 벡터, Big5 성향 호환성 |
| 키 | 키 차이나 키 선호가 중요한지 | 프로필 키 정보 기반 적합도 |
| 체형 | 체형 선호를 얼마나 볼지 | 프로필 체형 정보 기반 적합도 |
| 학교 | 같은 학교/학교 조건을 얼마나 볼지 | 학교 정보 기반 선호 또는 필터 가중치 |
| 취미 | 관심사가 비슷한지 | 취미/관심사 데이터가 있는 경우의 적합도 |
| 시간대 | 실제로 만날 시간이 잘 맞는지 | `available_timeslots` 교집합 적합도 |

주의:

- 외모 점수 원본이나 AI 점수 숫자는 사용자에게 노출하지 않는다.
- "외모"는 상대를 점수화해서 보여주는 값이 아니라, 내 선호와 상대의 특징이 얼마나 맞는지에 대한 내부 매칭 비중으로 설명한다.
- 취미 데이터가 아직 충분하지 않으면 UI에서는 "관심사" 정도로 표현하고, 실제 매칭 반영 강도는 엔진 구현 상태에 맞춰 조정한다.

## Recommended Default

기본값은 지금과 비슷하게 유지한다.

```json
{
  "appearance": 25,
  "personality": 25,
  "height": 10,
  "body_type": 10,
  "school": 10,
  "hobby": 10,
  "time_fit": 10
}
```

저장 시에는 기존 DB 계약에 맞게 0-1 소수로 변환한다.

```json
{
  "appearance": 0.25,
  "personality": 0.25,
  "height": 0.10,
  "body_type": 0.10,
  "school": 0.10,
  "hobby": 0.10,
  "time_fit": 0.10
}
```

## Proposed UI Components

### Replace

- `components/profile/PreferenceSliders.tsx`

### With

- `components/profile/PreferenceWeightInputs.tsx`

Props:

```ts
interface PreferenceWeightInputsProps {
  initialValue?: PreferenceWeights
  onChange: (weights: PreferenceWeights) => void
}
```

내부 상태는 퍼센트 정수로 관리한다.

```ts
type PreferenceWeightPercents = Record<keyof PreferenceWeights, number>
```

저장 직전 또는 `onChange` 직전에 100으로 나눠 기존 `PreferenceWeights` 형식으로 변환한다.

### Optional Visual Component

추가 컴포넌트:

- `components/profile/PreferenceRecipePreview.tsx`

역할:

- 현재 입력한 비율을 한눈에 보여주는 시각 요약.
- 각 항목 비율에 따라 색 면적 또는 작은 조각 크기가 달라진다.
- 실제 사람 사진은 사용하지 않는다.

예시:

```text
내 매칭 레시피
외모 25 · 성격 25 · 시간대 10 ...
```

구현은 SVG가 적합하다. 외부 이미지보다 실제 숫자에 따라 변하는 면적/조각을 만들기 쉽고, 텍스트 깨짐 문제도 없다.

생성 이미지를 쓰는 경우:

- 상단 배경/무드 이미지 1개만 사용한다.
- 각 항목별 이미지는 필요하면 나중에 추가한다.
- 생성 이미지 안에는 텍스트를 넣지 않는다.
- 저장 위치는 `public/profile/preferences/`가 적합하다.

## Validation Rules

- 각 값은 0 이상 100 이하.
- 소수점은 받지 않고 정수 퍼센트만 받는다.
- 빈 값은 임시로 0 처리하되, 입력 중에는 에러가 과하게 뜨지 않게 한다.
- 합계가 100이 아니면 저장 버튼 비활성화.
- 합계가 100이면 저장 가능 상태를 명확히 표시.
- 음수, 100 초과, 숫자 아닌 값은 입력 단계에서 보정하거나 저장 불가 처리.

## Interaction Details

### Quick Presets

상단에 프리셋을 두면 사용자가 시작하기 쉽다.

```text
균형형
외모 중시
성격 중시
시간 맞춤 중시
```

프리셋은 전체 값을 한 번에 바꾸는 명시적 행동이므로 괜찮다. 슬라이더처럼 몰래 다른 항목이 바뀌는 문제가 없다.

### Reset

`기본값으로 되돌리기` 버튼은 유지한다.

### Mobile

- 카드 내부에 숫자 입력을 크게 배치한다.
- `inputMode="numeric"` 사용.
- `+5`, `-5` 버튼은 44px 이상 터치 영역 확보.
- 합계 상태는 카드 목록 위에 sticky 또는 상단 고정 블록으로 둔다.

## Implementation Steps

1. `PreferenceSliders.tsx`를 새 컴포넌트로 교체하거나, 기존 파일명을 유지한 채 내부 UI를 숫자 입력 방식으로 바꾼다.
2. `ITEMS` 정의에 `helpText`를 추가한다.
3. 내부 상태를 0-1 소수 대신 0-100 정수로 관리한다.
4. `handleChange(key, percent)`는 해당 키만 수정한다.
5. `total`, `remaining`, `overage`를 계산해 상단에 표시한다.
6. `/profile/preferences/page.tsx`의 안내 문구를 숫자 입력 방식으로 변경한다.
7. 저장 로직은 기존 `preference_weights` JSONB 형태를 그대로 사용한다.
8. 필요하면 `docs/engineering/INTERFACE_CONTRACT.md`의 "슬라이더로 입력" 문구만 별도 PR에서 "UI에서 입력"으로 수정한다. 컬럼/타입은 바꾸지 않는다.
9. `PreferenceRecipePreview`를 추가해 사용자가 입력한 비율을 시각적으로 보여준다.
10. 필요 시 글자 없는 추상 무드 이미지를 생성해 상단 보조 비주얼로 사용한다.

## Open Questions

- 취미 데이터가 현재 매칭 엔진에서 충분히 반영되는지 확인 필요.
- 학교 가중치는 "같은 학교"인지 "학교/학과/캠퍼스 친숙도"인지 표현을 더 좁힐지 결정 필요.
- 숫자 입력만 둘지, 숫자 입력 + 작은 스텝 버튼을 둘지 디자인 확인 필요.
- 프리셋을 1차 구현에 넣을지, 숫자 입력 안정화 후 2차로 넣을지 결정 필요.

## Recommended Decision

1차 구현은 **숫자 입력 + 설명 + 합계 검증 + 기본값 초기화**까지만 한다.  
프리셋은 좋은 기능이지만, 먼저 사용자가 직접 가중치를 정확히 입력하는 기본 흐름을 안정화한 뒤 추가하는 편이 낫다.
