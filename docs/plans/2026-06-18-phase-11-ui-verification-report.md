# Phase 11 UI 검증 보고서

작성일: 2026-06-18
담당: 팀장방
범위: `/match`, `/match/dev-match-pending`, `/match/dev-match-1`, 홈-매칭 연결

## 1. 이번 검증의 기준

Phase 11의 목표는 기술적으로만 연결된 화면이 아니라, 사용자가 앱처럼 눌러보면서 다음 행동을 이해할 수 있는 흐름을 만드는 것이다.

이번 검증은 아래 기준으로 진행했다.

- 홈에서 오늘 해야 할 일이 보이는가
- `/match`에서 준비 중 매칭과 확정 매칭의 다음 행동이 구분되는가
- pending 매칭에서 사전 힌트 작성 UI가 한 항목씩 진행되는가
- 음식취향 카드에서 부산대 근처 예시와 지도 검색 흐름이 보이는가
- 논쟁카드가 여러 질문 A/B 선택 흐름으로 동작하는가
- confirmed 매칭에서 오늘 카드 뽑기와 채팅 진입이 보이는가
- DB/API 변경 없이 프론트에서 가능한 개선인지, 아니면 별도 DB/API 작업인지 분리되는가

## 2. 현재 확인된 동작

### `/match`

확인 결과:

- `매칭 허브` 화면이 보인다.
- pending 매칭 카드는 `준비 이어가기`로 표시된다.
- confirmed 매칭 카드는 `오늘 카드 확인`으로 표시된다.
- pending 카드는 `/match/dev-match-pending`으로 이동한다.
- confirmed 카드는 `/match/dev-match-1`로 이동한다.
- 상단에는 홈으로 돌아가는 링크가 있다.

판단:

- 홈-매칭 연결은 1차 앱 동선 기준으로 동작한다.
- 상태별 CTA 문구도 사용자가 이해 가능한 수준이다.

### `/match/dev-match-pending`

확인 결과:

- 매칭 상세에서 `사전 힌트 작성` 영역이 보인다.
- 카드 작성은 전체 펼침이 아니라 한 항목씩 이동하는 wizard 형태다.
- 현재 항목/완료 개수가 `1/6`, `1/6 완료`처럼 보인다.
- `다음 작성할 카드` 영역이 있다.
- `사전 힌트 저장`, `우리 그룹 무료 참여 완료`, `매칭 확정하기`는 조건 충족 전 비활성이다.

판단:

- 사용자가 카드 작성 흐름을 단계별로 눌러볼 수 있다.
- 다만 저장 가능 기준이 `4개`인데 화면 진행률은 `6개 항목`이라, 추후 문구를 더 직관적으로 다듬을 여지가 있다.

### 음식취향 카드

확인 결과:

- `다음` 버튼으로 음식점 카드까지 이동 가능하다.
- `부산대 근처 예시 3곳`이 보인다.
- 예시: `톤쇼우`, `겐쇼심야라멘`, `모모스커피`
- 추천 음식점을 누르면 입력값이 채워진다.
- 입력 후 `네이버지도에서 미리 보기` 링크가 나타난다.
- 링크는 `https://map.naver.com/p/search/부산대 <음식점명>` 형태다.

판단:

- 현재는 네이버지도 API/SDK 연동이 아니라 검색 링크 연동이다.
- Phase 11 범위에서는 적절하다.
- 실제 장소 식별, 좌표, 영업시간, 카테고리 정확도는 DB/API 작업으로 분리해야 한다.

### 논쟁카드

확인 결과:

- `인터넷 논쟁 카드` 항목으로 이동 가능하다.
- A/B 선택 카드가 보인다.
- 첫 질문에서 `찍먹 / 부먹` 선택지가 보인다.
- 선택하면 자동으로 `논쟁 2/4`로 넘어간다.
- 다음 질문에서 `민초 가능 / 반민초` 선택지가 보인다.

판단:

- 여러 질문을 순서대로 선택하는 구조는 동작한다.
- 선택 즉시 다음 질문으로 넘어가므로, 방금 고른 선택지를 잠깐 확인하는 느낌은 약하다.
- 이 부분은 프론트 UX 개선 후보지만, 필수 기능 차단은 아니다.

### `/match/dev-match-1`

확인 결과:

- 확정 매칭 상세가 보인다.
- `지도 열기` 링크가 보인다.
- `매칭 채팅방 보기` 링크가 보인다.
- `오늘의 공개 카드` 영역이 보인다.
- `오늘 카드 1번 뽑기`, `오늘 카드 2번 뽑기`, `오늘 카드 3번 뽑기` 버튼이 보인다.
- 카드 하나를 누르면 `공개 완료` 상태가 된다.

판단:

- 확정 후 오늘 카드 뽑기와 채팅 진입 흐름은 프론트 기준으로 동작한다.
- 실제 하루 1장 제한, 공개된 카드 내용의 원자적 저장은 DB/API 검증이 필요하다.

## 4. DB/API로 넘길 항목

아래는 이번 Phase 11 프론트 작업에서 바로 해결하지 않는 항목이다.

| 우선순위 | 항목 | 이유 |
| --- | --- | --- |
| P0 | 하루 1장 공개 카드가 통합 `content_text` 전체를 공개하지 않는지 검증 | 현재 작성 데이터가 항목별 DB 구조가 아니라 직렬화된 텍스트 중심이면 공개 단위가 섞일 수 있다. |
| P0 | confirmed 이후 시간/장소 배정 순서 검증 | 확정 뒤 실제 `match_meetings`, `venues` 생성이 누락되면 상세 화면의 약속 정보가 깨질 수 있다. |
| P1 | 음식점 seed/장소 DB | 지금은 네이버지도 검색 링크다. 실제 장소 선택, 좌표, 주소, 카테고리, 영업 상태는 별도 DB/API가 필요하다. |
| P1 | 친구별 카드 작성 완료 항목 상세 | 현재는 그룹 전체 완료 수 중심이다. 친구별로 어떤 항목이 비었는지 보려면 응답 구조가 더 필요하다. |
| P1 | 알림 DB 이벤트 | 홈/매칭 화면에서는 안내 가능하지만, 실제 카드 작성 요청/오늘 카드 가능 알림은 별도 이벤트 저장이 필요하다. |
| P2 | 오늘 카드 공개 가능 시간 실시간 갱신 | 16:00-20:00 정책은 DB 정책과 UI 상태를 같이 검증해야 한다. |

## 5. 검증 결과

명령 검증:

- `npm run typecheck`: 통과
- `npm run lint`: 통과
- `npm run test:matching`: 통과
- `npm run build`: 4분 제한에서 timeout

브라우저 검증:

- `http://localhost:3003/match`: 화면 렌더 및 pending/confirmed 카드 확인
- `http://localhost:3003/match/dev-match-pending`: 사전 힌트 wizard 확인
- `http://localhost:3003/match/dev-match-pending`: 음식점 카드, 추천 음식점, 네이버지도 검색 링크 확인
- `http://localhost:3003/match/dev-match-pending`: 논쟁카드 1/4에서 선택 후 2/4 이동 확인
- `http://localhost:3003/match/dev-match-1`: 오늘 카드 뽑기, 공개 완료, 채팅 링크 확인

주의:

- dev 서버 첫 라우트 컴파일이 느리다. 첫 요청은 10초 이상 걸릴 수 있다.
- `npm run build` timeout은 별도 원인 추적이 필요하다.
- `@컴퓨터` 직접 제어 도구는 현재 도구 목록에서 노출되지 않아 브라우저 검증으로 대체했다.

## 6. 팀장방 판단

Phase 11 프론트 UX의 핵심 흐름은 현재 로컬에서 확인 가능하다.

다만 목표 전체 완료 판정은 아직 보류한다. 이유는 다음과 같다.

- build timeout 원인 조사가 남아 있다.
- `@컴퓨터` 직접 검증은 도구 미노출로 수행하지 못했다.
- 카드 공개 단위와 장소 데이터는 DB/API 작업으로 분리되어야 한다.

다음 추천 작업:

1. Phase 11 변경분을 커밋하기 전, 프론트 변경 파일과 큐 조건 변경 파일을 분리해 커밋 범위를 정리한다.
2. `npm run build` timeout 원인을 별도 Phase로 추적한다.
3. DB/API 항목은 Bernoulli 작업으로 분리한다.
