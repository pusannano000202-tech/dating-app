# 2026-06-21 야간 사용자 흐름 점검 및 구현 계획

> 목적: 우리 앱 디자인 방향은 유지하면서, 실제 배포 사용자가 로그인부터 매칭 확정, 보증금, 데일리카드, 연락처 공개까지 자연스럽게 진행할 수 있는지 프론트와 백엔드를 함께 점검한다.
> 이번 문서는 실행 전 제출용 계획서다. 아직 코드 구현을 시작하지 않는다.

## 0. 계획서 자체 점검 결과

판단:

- 방향은 맞다.
- 다만 바로 장시간 자율 실행에 들어가기에는 보강이 필요했다.
- 이유는 범위가 로그인, 프론트, Supabase, 결제, 환불, 노쇼, 관리자까지 넓어서, 기준 없이 시작하면 토큰과 시간이 흩어진다.

이번 보강에서 추가한 안전장치:

- 먼저 감사 결과 문서를 만들고, 그 문서 없이 구현에 들어가지 않는다.
- 결정이 필요한 정책은 코드 수정하지 않는다.
- DB/API 변경 없이 가능한 프론트 수정만 첫 실행 범위로 둔다.
- 모든 파일을 넓게 읽지 않고, route/API/RPC 단위로 증거만 좁혀서 확인한다.
- 성공 기준과 중단 기준을 명확히 둔다.

## 0.5 이 문서의 정확한 용도

이 문서는 "몇 시간을 어떻게 쪼개서 쓸지"를 정하는 시간표가 아니다.

정확한 용도:

- 사용자가 자러 가기 전에 한 번 명령하면, Codex가 오래 붙잡고 진행할 수 있는 장시간 작업 패키지를 만든다.
- 실행자가 새 대화방이든 팀장방이든, 이 문서만 보고 같은 기준으로 움직이게 한다.
- 작업 시간이 길어지더라도, 무엇을 확인하고, 무엇을 고치고, 무엇은 멈춰서 물어봐야 하는지 정한다.
- 구현보다 먼저 `실제 사용자 동선`, `DB 저장 여부`, `결제 연결 상태`, `디자인 정리 기준`을 고정한다.

따라서 이번 단계의 성공 기준은 코드 수정량이 아니라 아래 산출물이 제대로 나오는 것이다.

- 사용자 전체 동선 감사 결과 문서.
- 결제/API 키 흡수 계획.
- 홈/매칭/그룹/알림 화면 역할 정리표.
- DB 저장 여부 체크리스트.
- 버튼을 눌렀을 때 화면이 어떻게 바뀌어야 하는지에 대한 점검표.
- 코드리뷰 결과와 다음 구현 우선순위.

## 1. 이번 작업의 기준

- 성준 디자인을 그대로 베끼지 않는다.
- 우리 버전의 따뜻한 크림톤, 코랄 포인트, 큰 카드형 화면, 하단 탭 구조를 기준으로 정리한다.
- 성준 작업에서는 좋은 흐름과 구조만 흡수한다.
- 성준 디자인에서 가져올 것은 색감, 여백, 카드 정돈감, 화면 위계, 버튼의 묵직함이다.
- 성준 디자인의 버튼 배치, 문구, 컴포넌트 코드를 통째로 복사하지 않는다.
- 사용자가 처음 들어와서 "뭘 누르면 되는지" 바로 알 수 있어야 한다.
- 실제 배포 기준으로 Supabase, 결제, 환불, 노쇼, 카드 공개가 어디까지 연결됐는지 증거로 확인한다.
- production Supabase는 건드리지 않는다.
- `lib/types.ts`, `supabase/migrations/`, `lib/supabase.ts`는 공용 영역이므로 수정 전 별도 보고한다.
- 화면에 보이는 말과 실제 DB 상태가 다르면 "완료"로 보지 않는다.
- 홈, 매칭, 그룹, 알림이 같은 말을 반복하면 중복으로 보고 줄인다.

## 2. 현재 확인된 핵심 충돌

### 2.1 보증금/환불/앱 기여 UX 충돌

현재 코드 기준:

- `lib/constants.ts`의 `DEPOSIT_AMOUNT`는 `10000`.
- `lib/payments/deposit.ts`는 성준 회신 반영 후 `mock`, `toss` provider만 가진다.
- 테스트도 10,000원 mock 보증금 기준을 고정하고 있다.

현재 사용자가 다시 정리한 방향:

- 기본 보증금은 10,000원 기준으로 본다.
- 만남/환불 단계에서 사용자가 앱에 얼마를 남길지 `0원~10,000원` 사이에서 자율 선택하게 한다.
- 선택 단위는 1,000원 단위가 자연스럽다.
- 프론트는 스크롤/슬라이더/스텝 버튼처럼 사용자가 올리고 내리며 고르는 형태가 좋다.
- 예전에 만든 "3,000원만요", "2,000원만요", "1,000원만요" 구걸 UX를 참고한다.
- 전액 환불도 가능해야 한다.

판단:

- 이제 쟁점은 `보증금 10,000원 vs 1,000원`이 아니다.
- 쟁점은 `10,000원 보증금`과 `0~10,000원 앱 기여/환불 선택`을 프론트와 DB에서 어떻게 분리할지다.
- 결제 금액, 환불 금액, 앱 기여 금액을 같은 값처럼 다루면 큰 버그가 난다.
- 계획서 기준으로는 `deposit_amount = 10000`, `user_selected_app_fee = 0~10000`, `refund_amount = 10000 - user_selected_app_fee` 구조를 우선 감사한다.

### 2.2 성격 기능 충돌

현재 코드 기준:

- `lib/matching/match-setup-status.ts`에서 매칭 준비 조건은 `personality`, `schedule`, `preferences`.
- `personality_preference_completed_at`이 없으면 매칭 준비 완료가 아니다.
- `preference_weights`도 `appearance`, `personality`, `height`, `body_type` 4개 기준이다.

사용자가 새로 말한 방향:

- "성격빼기"라고 적어둠.

판단:

- 성격 검사를 완전히 제거할지, 사용자 입력 부담만 줄이고 내부 매칭 요소로 유지할지 결정이 필요하다.
- 이번 계획에서는 "성격 완전 제거"가 아니라 먼저 "성격 입력이 실제로 필요한지, 화면에서 부담스럽게 보이는지"를 감사한다.

### 2.3 로그인/인증 충돌

현재 흐름:

- 이메일 OTP, Google OAuth 추가 코드가 있다.
- Supabase/Google/Vercel 대시보드 설정은 사용자가 직접 해야 하는 부분이 있다.
- AGENTS 문서에는 휴대폰 OTP라고 되어 있지만, 최근 작업 방향은 이메일 인증 중심이다.

판단:

- 코드와 대시보드 설정을 분리해서 봐야 한다.
- 앱 안의 로그인 버튼이 있어도 Supabase provider 설정과 redirect URL이 없으면 실제 배포에서는 실패한다.

### 2.4 프론트 상태 불일치

현재 증상:

- 홈과 매칭 화면의 "함께하는 친구" 정보가 서로 다르게 보인 적이 있다.
- `/profile/edit` 화면은 사용자가 보기에 개선이 반영되지 않은 것처럼 보였다.
- 홈에서 매칭을 찾기 전인데 상대 카드나 결과성 정보가 먼저 보이면 사용자 입장에서는 흐름이 이상하다.

판단:

- dev preview mock 상태와 실제 상태가 섞인 것이 원인일 가능성이 높다.
- "한 화면만 예쁘게"가 아니라 홈, 매칭, 그룹, 알림이 같은 상태 모델을 공유해야 한다.

### 2.5 성준 결제 API 키/API 흡수 범위

현재 코드 기준:

- 결제 provider 구조는 성준 회신 반영 후 `mock`, `toss`로 좁혔다.
- `DEPOSIT_AMOUNT`는 10,000원 기준이다.
- 테스트는 "실제 외부 결제사를 바로 호출하지 않고 준비 상태를 검증한다"는 쪽에 가깝다.
- 즉, 현재는 실결제 완성이라기보다 mock 보증금 기준선과 provider 연결 준비 단계로 봐야 한다.

성준 쪽 회신으로 확인된 기준:

- 결제사는 Toss 단일 기준이다. KakaoPay/PortOne은 현재 흡수 기본값이 아니다.
- 성준 결제 route/helper 기준은 `app/api/payments/{deposit,tip}/prepare`, `confirm`, `cancel`과 `lib/payments/{toss,orders,auth,config}.ts`다.
- 필요한 환경변수는 `NEXT_PUBLIC_TOSS_CLIENT_KEY`, `TOSS_SECRET_KEY`, `PAYMENT_INTERNAL_SECRET`, `SUPABASE_SERVICE_ROLE_KEY`다.
- 프론트 공개 키는 `NEXT_PUBLIC_TOSS_CLIENT_KEY`만 가능하고, `TOSS_SECRET_KEY`, `PAYMENT_INTERNAL_SECRET`, `SUPABASE_SERVICE_ROLE_KEY`는 서버에서만 써야 한다.
- 보증금 결제 성공 시 `deposits.status`가 `pending -> paid`로 바뀌고 `payment_key`, `method`, `paid_at`이 채워지는 구조다.
- 금액은 DB, 클라이언트 요청, Toss 승인 금액을 3중 검증해야 한다.
- 현재 webhook은 없고 동기 confirm 방식이다.
- 정상 만남 후 전액 환불은 내부 secret으로 `cancel` route를 호출하는 방향이며, 사용자 직접 호출은 막아야 한다.
- 만남 인증 이후 환불 호출 연결, 노쇼 판정, webhook은 아직 우리 구현 또는 정책 결정 대상이다.

계획서에 반영할 기준:

- API secret key는 절대 프론트에 넣지 않는다.
- `NEXT_PUBLIC_*`에는 공개 가능한 값만 둔다.
- 서버 route가 결제 승인/취소/webhook을 담당한다.
- 프론트 버튼은 결제창 이동, 결제 결과 복귀, 결제 완료 상태 표시까지만 담당한다.
- 결제 완료 판정은 프론트 URL query가 아니라 서버 검증과 DB 상태로 한다.
- 성준 API를 흡수해도 바로 production 결제가 되는 것은 아니다. Toss 테스트 상점 키, Vercel 환경변수 설정, confirm/cancel route 검증이 끝나야 실제 결제 검증이 가능하다.
- KakaoPay/PortOne 문구나 provider는 이번 흡수 작업의 기본 방향에서 제외한다. 현재 결제 provider 목록에서도 제거했고, 과거 문서는 레거시 참고로만 본다.

### 2.6 프론트 동선과 DB 저장 여부 충돌

현재 위험:

- 화면에는 "완료"처럼 보여도 실제로는 `localStorage`, `sessionStorage`, dev mock만 쓰는 부분이 섞일 수 있다.
- 사용자는 온보딩을 끝냈다고 느끼는데, 매칭 엔진이 읽는 `profiles`, `photos`, `available_timeslots`, `preference_weights`에는 값이 없을 수 있다.
- 매칭 찾기 버튼이 눌려도, 서버 gate가 필요한 정보를 못 찾으면 실제 배포에서는 막힌다.

점검 기준:

- 각 화면은 "입력 UI -> API 또는 Supabase 저장 -> DB 테이블/RPC -> 다음 화면 반영"까지 연결됐는지 확인한다.
- 저장 근거가 없는 화면은 `mock`, `로컬전용`, `DB 미연결`로 표시한다.
- 사용자에게 중요한 입력값은 화면 문구만 고치지 말고, 실제 저장 위치를 확인한다.

특히 확인할 데이터:

- 이메일/Google auth user.
- 이름, 닉네임, 전화번호, 나이, 학과.
- 이상형 월드컵 결과와 선택 로그.
- 본인 성격 또는 제거 여부.
- 선호 성향.
- 사진과 Storage URL.
- 가능 시간.
- 매칭 비중 4개. 단, 성준 회신의 7개 가중치 기준과 충돌하므로 계약 합의 전에는 공용 타입/DB를 바꾸지 않는다.
- 친구 요청과 그룹 멤버 상태.
- 보증금 결제 상태.
- 데일리카드 작성/공개 상태.
- 연락처/채팅 공개 상태.

### 2.7 홈/매칭/그룹/알림 화면 역할 중복

현재 문제:

- 홈에 있는 내용이 매칭에도 있고, 매칭에 있어야 할 내용이 홈에도 섞여 있다.
- 사용자는 "지금 내가 무엇을 해야 하는지"보다 "비슷한 카드가 여러 군데 보인다"는 느낌을 받을 수 있다.
- 같은 그룹 상태가 화면마다 다르게 보이면 신뢰가 깨진다.

역할 재정의:

- 홈(`/`): 오늘 해야 할 다음 행동 1개, 앱의 짧은 설명, 현재 진행 중인 매칭 요약만 보여준다.
- 매칭(`/match`): 매칭 준비 상태, 현재 그룹, 부족한 입력, 매칭 찾기 버튼, 큐/결과 상태를 보여준다.
- 그룹(`/group/create`): 친구 초대, 멤버 수락, 준비 완료 여부를 보여준다.
- 알림(`/notifications`): 매칭 도착, 카드 공개, 약속 임박처럼 행동이 필요한 이벤트를 카드로 보여준다.
- 내정보(`/profile/*`): 프로필/사진/시간/가중치/선호 성향을 수정하는 곳으로 제한한다.

정리 기준:

- 홈에는 상대 카드 상세를 먼저 노출하지 않는다.
- 매칭을 찾기 전에는 "추천 상대팀"을 결과처럼 보여주지 않는다.
- 그룹 멤버 정보는 홈/매칭/그룹이 같은 source를 쓰도록 한다.
- "How it works"는 큰 반복 섹션이 아니라 우측 상단 정보 버튼 또는 짧은 도움말로 낮춘다.

### 2.8 부산대 학과 검색 입력 누락

현재 문제:

- 기본정보 화면의 학과 입력이 사용자가 보기에 단순 입력칸처럼 보인다.
- 사용자가 이전에 제공한 부산대학교 단과대/학과 목록 기준으로 검색하고 고르는 구조가 필요하다.
- 현재 `lib/pnu-departments.ts`는 존재하지만, 사용자가 다시 제공한 목록보다 짧고 일부 학과명이 최신 기준과 다를 수 있다.

사용자 기준:

- 단과대별로 그룹화된 부산대 학과 목록을 source of truth로 둔다.
- 학과 입력은 직접 타이핑만 하는 방식이 아니라 검색하면 후보가 주르륵 뜨는 방식으로 한다.
- 예: "컴" 입력 시 `컴퓨터공학전공`, `정보컴퓨터공학부` 등 관련 후보가 뜬다.
- 선택된 값은 `profiles.department`에 저장되어야 한다.

계획 반영:

- `lib/pnu-departments.ts`를 사용자가 제공한 단과대/학과 목록과 비교한다.
- 누락/오래된 명칭을 표로 정리한다.
- 구현 단계에서는 grouped searchable picker를 유지하거나 보강한다.
- 검증 단계에서는 실제 화면에서 검색, 선택, 저장, 재진입 시 값 유지까지 확인한다.

### 2.9 성별에 따른 이상형 월드컵 후보

현재 점검해야 할 문제:

- 사용자가 `여자`로 입력하면 남자 후보가 떠야 한다.

계획 반영:

- `/profile/basic` 저장 후 `/profile/worldcup` 진입 시 성별을 어디서 읽는지 확인한다.
- dev preview fallback이 실제 Supabase 모드와 다르게 성별을 고정하지 않는지 확인한다.
- 월드컵 이미지 metadata가 남자/여자 후보를 구분하는지 확인한다.
- 성별 정보가 없을 때는 월드컵으로 바로 보내지 말고 기본정보 보완 화면으로 보내는 것이 맞는지 점검한다.

### 2.10 나이 비슷한 사람끼리 매칭되는지

현재 점검해야 할 문제:

- 프론트에서 "비슷한 나이끼리 매칭"이라고 말한다면 실제 매칭 엔진 또는 서버 gate도 그 기준을 써야 한다.
- 기본정보에서 나이를 입력해도 매칭 점수/필터에 반영되지 않으면 사용자에게 거짓 기대를 준다.

계획 반영:

- `profiles.age`, `profiles.preferred_age_min`, `profiles.preferred_age_max`가 실제로 저장되는지 확인한다.
- 매칭 점수 계산 또는 매칭 엔진 입력에서 나이를 읽는지 확인한다.
- 비슷한 나이 기준을 `동갑/+-1세/+-2세/나이대` 중 무엇으로 표현할지 정리한다.
- 프론트에서는 "비슷한 나이"를 말하기 전에 실제 적용 기준을 보여준다.
- 나이 입력 UI도 학과처럼 사용자가 헷갈리지 않게 범위/선택형으로 정리할 수 있는지 검토한다.

### 2.11 성준 회신 반영: 스키마 기준 충돌

성준 회신 핵심:

- 성준 현재 `dating-app` 본체 기준으로 확실히 굳은 DB 레이어는 Toss 보증금 결제 쪽이다.
- 성준 기준에는 `match_meetings`, `scheduled_start`, `venues`, `map_url`, `enter_match_pool`, `preferred_personality_vector`, `member_match_setup_ready`, `current_user_match_setup`, `connections` RPC/테이블, `16~20시 직접뽑기`가 없다고 회신했다.
- `gwating-app` 스냅샷에는 매칭 결과, 일정 조율, 데일리 Q&A, 당일 채팅 UX가 localStorage/mock 기반으로 구현되어 있다.
- 하지만 `gwating-app`은 Supabase/RPC와 연결된 실제 백엔드가 아니다.
- 성준은 `preference_weights`를 7개 키(`appearance`, `personality`, `height`, `body_type`, `school`, `hobby`, `time_fit`)로 본다고 했다.
- 현재 우리 로컬 `docs/engineering/INTERFACE_CONTRACT.md`와 구현 흐름은 4개 키(`appearance`, `personality`, `height`, `body_type`)를 기준으로 적혀 있다.

판단:

- 이건 단순 충돌 해결 문제가 아니라 기준 문서와 실제 성준 브랜치의 레이어가 어긋난 상태다.
- 우리 브랜치에 `match_meetings`, `venues`, `enter_match_pool`, `connections` 계열 코드가 있더라도, 성준 쪽 현재 기준에는 대응물이 없다고 봐야 한다.
- 따라서 새 migration이나 공용 타입 변경으로 밀어붙이면 "성준 스키마 + 우리 스키마"가 따로 생기는 이중 스키마가 된다.
- 이 단계에서는 기존 코드를 삭제하지 않고, `우리 브랜치 구현 후보`, `성준 결제 기준`, `gwating-app 프로토타입`, `합의 필요 스키마`로 분리한다.

작업 기준:

- `supabase/migrations/`, `lib/types.ts`, `docs/engineering/INTERFACE_CONTRACT.md`는 바로 수정하지 않는다.
- `preference_weights` 4개/7개 문제는 코드 수정 전에 성준과 계약 결정을 받아야 한다.
- 데일리카드 16~20시 직접 뽑기와 gwating 자동 분배 방식은 정책 충돌로 분류한다.
- `gwating-app`의 Q&A/일정/채팅은 UX 참고 대상으로만 본다. Supabase 저장이 붙기 전에는 실제 구현 완료로 부르지 않는다.
- 결제 흡수는 Toss 단일, 10,000원 보증금, 내부 secret 환불 호출 기준으로 좁힌다.

### 3.1 DB 저장 여부 판정 방식

각 화면은 아래 5단계로 판단한다.

| 판정 | 의미 | 예시 |
| --- | --- | --- |
| `DB 연결 완료` | 사용자가 입력한 값이 Supabase table/RPC에 저장되고 다음 화면에서 다시 읽힌다 | `profiles.preference_weights` 저장 후 매칭 준비 조건에 반영 |
| `API 있음, 검증 부족` | route/API는 있으나 실제 저장 성공/실패 UI가 불명확하다 | 결제 confirm route는 있으나 외부 결제사 승인 호출 확인 필요 |
| `로컬전용` | `localStorage`, `sessionStorage`, dev cookie만 쓴다 | preview용 카드 작성 상태 |
| `화면만 있음` | UI는 있는데 저장 route 또는 DB 근거가 없다 | 결과처럼 보이는 mock 카드 |
| `정책 결정 필요` | 구현 전에 금액, 공개 시점, 제거 여부 같은 결정이 필요하다 | 앱 기여금 0~10,000원 선택, 성격 제거 |

감사 문서에는 반드시 아래 형태로 남긴다.

```markdown
사용자 행동: 매칭 비중 입력
화면: /profile/preferences
저장 위치: profiles.preference_weights
다음 화면 반영: /match/start gate, /match
현재 판정: DB 연결 완료 또는 부분완료
문제: 합계 검증/문구/버튼 상태 등
다음 조치: 프론트 수정 또는 DB/API 수정
```

### 3.2 성준 디자인 흡수 판정 방식

가져올 것:

- 전체 색감의 정돈감.
- 카드의 깊이감과 여백.
- 하단 탭의 명확한 현재 위치 표시.
- 큰 CTA 버튼의 묵직한 느낌.
- 매칭 진행률과 상태 배지의 직관성.
- "지금 할 일"이 한눈에 보이는 화면 구조.

가져오지 않을 것:

- 성준 화면의 컴포넌트 코드 통째 복사.
- 우리 앱과 맞지 않는 검은색/강한 대비 과사용.
- 같은 기능을 홈과 매칭에 중복 노출하는 구조.
- 실제 DB 상태와 다른 mock 상태를 결과처럼 보여주는 방식.
- 사용자가 아직 하지 않은 행동을 이미 완료된 것처럼 보이게 하는 문구.

우리 기준:

- 홈은 부드럽고 간단하게.
- 매칭은 진행 상태가 명확하게.
- 그룹은 친구와 함께 들어가는 느낌이 명확하게.
- 결제/보증금은 불안하지 않게, 환불 조건을 짧고 정확하게.
- 알림은 "읽을거리"가 아니라 "눌러야 할 행동"으로 보이게.

### 3.3 버튼/화면 전환 점검표

장시간 작업자는 아래 버튼을 실제 사용자 관점에서 눌러보고, 화면 변화가 자연스러운지 기록한다.

| 화면 | 버튼/행동 | 기대되는 화면 변화 |
| --- | --- | --- |
| `/login` | 이메일 로그인 | 메일 인증 안내 후 callback 성공 시 온보딩 다음 단계로 이동 |
| `/login` | Google 로그인 | Supabase OAuth callback 후 원래 가려던 경로 또는 온보딩으로 이동 |
| `/profile/basic` | 학과 검색 | 입력한 글자에 맞는 부산대 학과 후보가 뜨고 선택하면 칸에 반영 |
| `/profile/basic` | 저장하고 이상형 월드컵으로 | DB 저장 성공 후 성별에 맞는 월드컵 후보 화면으로 이동 |
| `/profile/worldcup` | 후보 선택 | 다음 라운드로 자연스럽게 전환, 마지막에는 결과 저장 |
| `/profile/survey` | 성향 저장 | 저장 후 다음 온보딩 단계 또는 매칭 준비 화면으로 이동 |
| `/profile/schedule` | 가능 시간 저장 | 저장 후 매칭 준비 조건이 완료 상태로 바뀜 |
| `/profile/preferences` | 매칭 비중 저장 | 4개 가중치 합계가 맞고 저장 후 매칭 찾기 조건에 반영 |
| `/friends` | 닉네임 친구 요청 | 존재하는 닉네임이면 요청 전송, 없는 닉네임이면 찾을 수 없음 안내 |
| `/group/create` | 친구 초대 | 미가입 즉시 합류처럼 보이지 않고 가입/수락 대기 상태로 표시 |
| `/match` | 매칭 찾기 | 부족한 조건이 있으면 막고, 모두 완료되면 큐/가매칭 단계로 이동 |
| `/match/[id]` | 카드 작성 | 내 카드 항목 완료율이 올라가고 상대 공개 조건이 설명됨 |
| `/match/[id]` | 보증금 결제 | 결제 진행/완료/실패 상태가 명확히 바뀜 |
| `/match/[id]/refund` | 앱 기여금 조절 | 0~10,000원, 1,000원 단위로 움직이고 환불액이 즉시 계산됨 |
| `/match/[id]/refund` | 전액 환불 선택 | 3,000원만요 -> 2,000원만요 -> 1,000원만요 식의 설득 UX가 뜨되 강제하지 않음 |
| `/notifications` | 매칭 도착 카드 | 매칭 상세 또는 해야 할 행동 화면으로 이동 |

## 4. 작업 단계

### Phase A. 증거 수집 감사

목표:

- 현재 코드가 실제로 무엇을 하는지 먼저 확인한다.
- "되어 있는 것처럼 보이는 UI"와 "DB까지 연결된 기능"을 분리한다.

작업:

- `git status --short --branch`로 현재 변경분을 분류한다.
- 위 표의 주요 파일을 읽고 API, RPC, Supabase 저장 위치를 확인한다.
- `docs/plans/2026-06-21-overnight-user-flow-audit-result.md`에 감사 결과를 작성한다.

산출물:

- 구현 상태 표.
- mock/미구현/결정필요 목록.
- 바로 고칠 프론트 문제 목록.

Phase A 완료 조건:

- 아래 5개 분류가 모두 채워져야 한다.
  - 실제 구현됨
  - 부분 구현됨
  - 로컬/mock만 있음
  - 미구현
  - 사용자 결정 필요
- 각 분류에는 파일 경로 또는 API/RPC 근거가 최소 1개 이상 있어야 한다.
- 근거 없이 "되어 있을 것 같다"라고 쓰지 않는다.

### Phase B. 사용자 체감 프론트 동선 정리

목표:

- 배포 전 사용자가 눌러보고 싶은 흐름으로 화면을 정리한다.
- DB/API 변경 없이 가능한 것부터 한다.

수정 후보:

- `/profile/edit`: 현재 사용자가 "안 바뀐 것 같다"고 느낀 화면을 우리 디자인 기준으로 정리.
- `/`: 홈은 앱 설명과 다음 행동 중심으로 정리. 중복된 매칭 상태는 줄인다.
- `/match`: 현재 그룹, 부족한 준비 항목, 매칭 찾기 버튼 상태를 한눈에 보이게 정리.
- `/group/create`: 친구가 들어오고 그룹이 만들어지는 느낌을 직관화.
- `/notifications`: "매칭이 도착했어요" 카드가 실제 매칭 카드처럼 보이도록 정리.
- `HomeInfoButton`: "How it works"는 큰 섹션보다 우측 상단 i 버튼/모달 방식으로 낮춘다.

원칙:

- 상대 카드나 결과성 정보는 매칭 찾기 전 홈에서 과하게 노출하지 않는다.
- "함께하는 친구" 정보는 홈과 매칭 화면이 같은 mock/source를 쓰게 한다.
- 미가입 친구를 바로 그룹에 넣는 듯한 표현은 제거하고, 초대 링크로 가입 후 참여하는 흐름으로 설명한다.
- 전화번호 친구 요청 표현은 닉네임 친구 요청 중심으로 바꾼다.

### Phase C. 온보딩과 Supabase 저장 흐름 정리

목표:

- 사용자가 회원이 되는 순간 어떤 정보가 Supabase에 들어가는지 명확히 한다.

점검 순서:

1. `/login`: auth user 생성 확인.
2. `/profile/basic`: `profiles`, `users.phone`, `profiles.age`, `profiles.gender`, `profiles.department` 저장 확인.
3. `/profile/basic`: 부산대 학과 검색/선택 UI가 사용자가 제공한 목록 기준인지 확인.
4. `/profile/worldcup`: 기본정보의 성별에 맞는 후보가 뜨는지 확인.
5. `/profile/worldcup`: 이상형 결과와 로그 저장 확인.
6. `/profile/survey`: Big5 저장 확인.
7. `/profile/photos`: Storage와 `photos` 저장 확인.
8. `/profile/schedule`: 가능 시간 저장 확인.
9. `/profile/preferences`: 매칭 비중 저장 확인.

결과 판단:

- 저장됨: 실제 Supabase row가 바뀌는 기능.
- 로컬전용: `sessionStorage`, `localStorage`, dev cookie만 쓰는 기능.
- 미연결: 화면만 있고 DB 저장이 없는 기능.
- UX 불일치: 저장은 되지만 사용자가 보기에는 선택/검색/전환이 불편한 기능.

### Phase D. 매칭 찾기 gate 정리

목표:

- 친구/그룹/성향/가능시간/매칭비중이 준비되지 않으면 매칭 찾기를 못 누르게 한다.
- 막는 대신 무엇을 하면 되는지 화면에서 바로 안내한다.

확인할 조건:

- 그룹 인원 충족.
- 친구 초대 수락.
- 각 멤버 준비 완료.
- 가능 시간 입력.
- 매칭 비중 입력.
- 성향 입력 유지 여부.
- 사전 카드 입력 필요 시점.
- 보증금 결제 필요 시점.

주의:

- 현재 코드상 성향 선호가 필수 조건이면, "성격빼기" 결정 전까지 임의 제거하지 않는다.
- 제거할 경우 `lib/matching/match-setup-status.ts`, `app/match/start/page.tsx`, UI 문구, 테스트를 함께 바꿔야 한다.

### Phase E. 보증금/결제/환불 정책 정리

목표:

- 사용자가 말한 "전액 환불 보증금" 정책을 실제 결제 흐름과 맞춘다.

먼저 결정할 것:

- 보증금 기본액: 현재 10,000원 기준을 유지할지 확인.
- 앱 기여금: 환불 단계에서 사용자가 0~10,000원 사이를 1,000원 단위로 고르게 할지 확인.
- 결제 provider: 성준 회신 기준 Toss 단일을 우선한다. 기존 `kakao`/PortOne 관련 provider는 현재 코드 기준에서 제외했고, 남아 있는 과거 문구는 문서 레거시로만 분류한다.
- 전액 환불 조건: 정상 만남 완료 시 즉시 환불인지, 관리자 확인 후 환불인지.
- 노쇼 처리: GPS 체크인, 인증사진, 관리자 판정 중 무엇을 MVP로 둘지.
- 구걸 UX: 3,000원만요, 2,000원만요, 1,000원만요 식의 단계형 설득을 살릴지 확인.

현재 코드 기준 후보:

- `mock` 결제는 즉시 paid 상태를 만들 수 있다.
- `mock`, `toss` provider readiness 구조만 유지한다. 실제 흡수 대상은 Toss이고, `mock`은 로컬/검토용이다.
- 환불 API와 refund page는 존재하지만 정책 문구와 금액 기준 재확인이 필요하다.
- `lib/refund/fee-flow.ts`와 환불 테스트가 예전 구걸/앱 기여 UX와 맞는지 확인해야 한다.

수정 예상 파일:

- `lib/constants.ts`
- `lib/payments/deposit.ts`
- `lib/refund/fee-flow.ts`
- `app/api/deposits/route.ts`
- `app/api/payments/deposit/*`
- `app/api/matches/[id]/refund/route.ts`
- `app/match/[id]/refund/page.tsx`
- `tests/config/deposit-policy.test.ts`
- `tests/config/deposit-payment-routes.test.ts`
- 필요 시 새 migration

환불/앱 기여 프론트 UX 기준:

- 결제한 보증금은 10,000원으로 표시한다.
- 만남 후 환불 화면에서 사용자가 앱에 남길 금액을 `0원, 1,000원, 2,000원 ... 10,000원` 중 선택한다.
- UI는 슬라이더, 스텝 버튼, 스크롤 선택 중 모바일에서 가장 조작이 쉬운 방식으로 설계한다.
- 사용자가 0원을 선택하면 전액 환불이 가능해야 한다.
- 전액 환불을 누르면 예전 구걸 UX를 참고해 `3,000원만 남겨주실래요?`, `2,000원만요`, `1,000원만요` 같은 단계형 제안을 보여준다.
- 사용자가 계속 전액 환불을 원하면 막지 않는다.
- 화면에는 `앱 기여금`, `환불 예정 금액`, `실제 보증금`을 분리해서 보여준다.
- DB/API도 `deposit amount`, `app fee`, `refund amount`를 분리해서 확인한다.

성준 결제 API 흡수 작업 순서:

1. 성준 브랜치 또는 handoff에서 결제 provider와 API key 이름을 확인한다.
2. 현재 코드의 env 이름과 성준 env 이름을 표로 맞춘다.
3. secret key는 server route에서만 읽게 한다.
4. public/client key만 프론트에 전달한다.
5. `/api/payments/deposit`에서 결제 요청을 만들고 결제창으로 이동할 정보를 반환한다.
6. `/api/payments/deposit/confirm`에서 결제사 승인값을 서버에서 검증한다.
7. 검증 성공 시 `deposits.status = paid`, `paid_at`, `provider`, `provider_payment_id` 계열 값을 저장한다.
8. `/api/payments/deposit/cancel` 또는 refund route에서 환불/취소를 처리한다.
9. webhook route는 결제사 이벤트를 받아 DB 상태와 불일치가 없는지 맞춘다.
10. 프론트는 `결제 전`, `결제 진행 중`, `결제 완료`, `환불 대기`, `환불 완료`, `실패` 상태를 명확히 보여준다.

사용자가 직접 해야 할 가능성이 높은 결제 설정:

- 결제사 테스트 상점 생성.
- 테스트 secret key/client key 발급.
- Vercel 환경변수 등록.
- Supabase/Vercel 배포 URL 기준 webhook URL 등록.
- 실제 결제 전 사업자/정산/환불 정책 확인.

Codex가 해야 할 결제 작업:

- env 이름 통일.
- 키 누락 시 읽기 쉬운 에러 문구.
- mock provider와 real provider 분리.
- 결제 시작/승인/취소/webhook route 정리.
- 프론트 결제 버튼과 상태 표시 연결.
- 결제 테스트는 외부 API를 직접 때리지 않고 provider client를 mock해서 작성.

### Phase F. 데일리카드와 연락처 공개 흐름 정리

목표:

- 매칭이 잡힌 뒤, 하루에 한 장씩 직접 뽑는 느낌을 명확히 한다.
- 내가 쓴 만큼 상대 카드가 열리는 구조를 사용자에게 납득시킨다.

정리할 정책:

- "내 카드 4/6 완료"와 "그룹 3명 중 2명 작성 완료"를 분리.
- 인터넷 논쟁카드는 여러 질문 선택형으로 유지.
- 데일리카드는 16:00~20:00 직접 뽑기.
- 매칭 당일 또는 약속 확정 시 전화번호와 채팅이 열리는 조건 확인.

수정 예상 파일:

### Phase G. 관리자 검토와 노쇼 처리

목표:

- "직접 사진 보고 매칭이 적합한지 판단하는 관리자 앱"이 실제로 어디까지 가능한지 확인한다.
- 노쇼 보증금 환불 문제를 사진/시간/장소 인증 흐름으로 풀 수 있는지 계획한다.

- 출석 상태 API, 체크인 API, 노쇼 확정 API가 존재한다.

### Phase H. 배포 설정과 외부 서비스 연결

목표:

- 코드가 있어도 대시보드 설정이 빠져서 실제 로그인/결제가 실패하는 상황을 막는다.

사용자가 직접 해야 할 가능성이 높은 것:

- Supabase Google Auth provider 활성화.
- Google Cloud OAuth client 생성.
- Supabase redirect URL allow list 등록.
- Vercel 환경변수 등록.
- Toss 테스트 상점 설정.
- 결제 webhook URL 등록.

Codex가 할 수 있는 것:

- 필요한 환경변수 목록 정리.
- 코드에서 env 이름 통일.
- 설정 누락 시 사용자에게 읽히는 에러 문구 개선.
- 로컬 mock과 production provider를 분리.

### Phase I. 전체 코드리뷰

목표:

- 지금까지 급하게 붙인 프론트, mock, 결제, DB 흐름이 서로 꼬이지 않았는지 확인한다.
- "작동하는 것처럼 보이는 화면"과 "실제 배포 가능한 코드"를 분리한다.

리뷰 항목:

- dirty 작업트리에 어떤 변경이 남아 있는지.
- 홈/매칭/그룹/알림이 같은 데이터를 서로 다르게 보여주는지.
- `localStorage`, `sessionStorage`, dev bypass가 사용자 흐름에 섞여 있는지.
- `lib/constants.ts`의 보증금 금액과 화면 문구, migration, 테스트가 일치하는지.
- 결제 secret key가 프론트로 새는 구조가 없는지.
- `lib/types.ts`, `lib/supabase.ts`, `supabase/migrations/` 같은 공용 파일 변경이 필요한지.
- DB 저장이 필요한 프론트 입력이 실제 저장 없이 화면만 있는지.
- 매칭 찾기 gate가 프론트와 서버에서 같은 조건을 쓰는지.
- 성준 담당 영역과 충현 담당 영역이 섞여 충돌 위험이 생기는지.
- 테스트가 현재 정책을 고정하고 있는지, 과거 정책을 붙잡고 있는지.

산출물:

- `docs/plans/2026-06-21-overnight-code-review-result.md`
- 우선순위별 문제 목록.
- 바로 고칠 수 있는 프론트 문제.
- DB/API 변경이 필요한 문제.
- 성준과 같이 결정해야 하는 문제.

### Phase J. 팀장방/Codex가 해야 할 일

팀장방 역할:

- 사용자의 목표를 기능 단위로 쪼갠다.
- 성준 작업과 우리 작업이 겹치는 파일을 먼저 분리한다.
- 구현 전에 dirty 작업트리를 분류한다.
- 공용 파일 수정은 바로 하지 않고 먼저 보고한다.
- 프론트 수정과 DB/API 수정 명령을 분리한다.
- 실행 결과를 문서로 남겨 새 대화방에서도 이어갈 수 있게 한다.

이번 장시간 실행 패키지에서 팀장방이 직접 해야 할 일:

- 이 계획서를 기준으로 감사 결과 문서를 먼저 만든다.
- 결제 키/API 흡수는 코드 구현 전에 성준 handoff와 현재 payment 코드의 차이를 표로 만든다.
- 홈/매칭/그룹/알림의 역할표를 만든다.
- DB 저장 여부 체크리스트를 완성한다.
- 바로 수정 가능한 프론트 항목과 사용자 결정이 필요한 항목을 분리한다.
- 마지막에 다음 실행 명령을 한 줄로 제안한다.

팀장방이 하지 말아야 할 일:

- 계획 승인 없이 장시간 구현을 바로 시작하지 않는다.
- 결제 금액을 임의로 바꾸지 않는다.
- 성격 기능을 임의로 제거하지 않는다.
- 성준 디자인을 통째로 복사하지 않는다.
- production Supabase나 실제 결제 상점 설정을 건드리지 않는다.

## 5. 장시간 자율 실행 작업 묶음

이 섹션은 시간표가 아니다.
사용자가 "이 계획대로 진행해"라고 명령하고 자리를 비웠을 때, Codex가 오래 붙잡고 진행할 작업 순서다.

### 묶음 1. 현재 상태 고정

- git 상태를 분류한다.
- 현재 브랜치와 원격 차이를 확인한다.
- dirty 작업트리를 `프론트`, `결제`, `문서`, `DB`, `기타`로 분류한다.
- 성준 handoff/payment 관련 문서와 현재 코드의 기준점을 확인한다.
- 이미 작업 중인 변경을 덮어쓰지 않는다.

### 묶음 2. 사용자 동선과 DB 저장 감사

- 로그인부터 매칭 확정까지 "화면 -> 저장 -> 다음 화면 반영" 흐름을 작성한다.
- 주요 route/API/migration 근거를 확인한다.
- 구현/부분/mock/미구현/결정필요 표를 작성한다.
- 실제 DB에 저장되지 않는 화면을 따로 표시한다.
- `localStorage`, `sessionStorage`, dev bypass만 쓰는 기능은 배포 가능 기능으로 보지 않는다.

### 묶음 3. 부산대 기본정보 UX 감사

- 사용자가 제공한 부산대 단과대/학과 목록과 `lib/pnu-departments.ts`를 비교한다.
- 학과 검색이 단과대 그룹/부분 검색/선택/저장/재진입 유지까지 되는지 확인한다.
- 나이 입력이 실제 매칭 조건이나 선호 나이에 반영되는지 확인한다.
- 기본정보의 성별이 이상형 월드컵 후보 성별을 바꾸는지 확인한다.

### 묶음 4. 결제/환불/앱 기여금 감사

- 기본 보증금 10,000원 기준이 코드/테스트/문구/DB에서 일치하는지 확인한다.
- 환불 화면에서 `앱 기여금 0~10,000원`, `1,000원 단위 선택`, `환불 예정 금액`을 구현할 수 있는지 확인한다.
- 예전 구걸 UX인 `3,000원만요`, `2,000원만요`, `1,000원만요` 흐름이 남아 있는지 찾는다.
- 성준 결제 API 키/결제사/handoff 기준을 확인한다.
- env 이름, server route, frontend button, DB 상태 변경, webhook을 연결 표로 작성한다.
- 사용자가 대시보드에서 해야 하는 일과 Codex가 코드로 할 일을 분리한다.

### 묶음 5. 프론트 동선과 디자인 정리안 작성

- 홈, 매칭, 그룹, 알림, 내정보의 역할을 재정의한다.
- 서로 다른 mock 상태를 하나의 dev preview source로 묶는 계획을 세운다.
- 매칭 찾기 전 상대 카드 노출 여부를 정리한다.
- 성준 디자인에서 가져올 색감/깔끔함/위계 기준을 작성한다.
- 중복 카드와 중복 문구 제거 후보를 작성한다.
- 버튼을 눌렀을 때 화면이 어떻게 바뀌어야 하는지 점검표를 채운다.

### 묶음 6. 전체 코드리뷰

- 공용 파일 변경 위험을 확인한다.
- 결제/환불/노쇼/데일리카드/채팅의 end-to-end 연결 상태를 확인한다.
- 매칭 찾기 gate가 프론트와 서버에서 같은 조건을 쓰는지 확인한다.
- "비슷한 나이", "성별에 맞는 이상형", "전액 환불 가능"처럼 화면에서 말한 약속이 실제 코드에 있는지 확인한다.
- 테스트가 현재 정책을 제대로 고정하는지 확인한다.
- 코드리뷰 결과 문서를 작성한다.

### 묶음 7. 보고와 다음 실행 명령 정리

- 감사 결과를 요약한다.
- 사용자 결정 필요 항목을 정리한다.
- 바로 구현 가능한 프론트 항목을 정리한다.
- DB/API 변경이 필요한 항목을 정리한다.
- 다음 명령어 1~3개를 제안한다.

주의:

- 이 작업 묶음은 "무조건 구현"이 아니라 "감사 + 실행 설계 + 안전한 프론트 후보 분리"가 중심이다.
- 승인 없이 DB/API는 바꾸지 않는다.
- `supabase/migrations/`, `lib/types.ts`, `lib/supabase.ts`는 건드리지 않는다.
- 보증금 기본액, 성격 제거, 실제 결제 provider 연결은 정책 결정 전에는 구현하지 않는다.

- 새 DB migration 추가.

## 6.5 실행 중단 기준

아래 상황이 나오면 계획을 계속 밀지 말고 멈춰서 보고한다.

- 보증금 금액을 바꿔야만 다음 작업이 가능한 경우.
- 성격 기능 제거 여부를 결정해야만 다음 작업이 가능한 경우.
- 새 migration 없이는 해결할 수 없는 경우.
- production Supabase 설정이나 Vercel 대시보드 설정이 필요한 경우.
- 현재 dirty 작업트리에서 다른 사람 변경과 충돌 위험이 큰 경우.
- 같은 화면을 고쳤는데 로컬에서 계속 이전 화면이 보이는 경우. 이때는 구현보다 서버/캐시/브랜치 상태를 먼저 보고한다.

## 6.6 토큰과 시간 절약 방식

토큰이 무제한이 아니므로 아래 방식으로 한다.

- 모든 파일을 처음부터 끝까지 읽지 않는다.
- 먼저 `rg`로 route, API, RPC, table 이름을 찾는다.
- 근거가 필요한 파일만 연다.
- 한 번에 너무 넓은 검색을 하지 않는다.
- 스크린샷 검증은 핵심 route만 한다.
- 문서에는 긴 코드 설명을 붙이지 않고, 파일 경로와 판단만 남긴다.
- 40분 단위로 현재까지 얻은 결론을 문서에 반영한다.
- 새 기능을 많이 만들기보다, 배포 전 동선의 막힘과 정책 충돌을 확실히 분리한다.
