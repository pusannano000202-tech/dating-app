# Manus 인수인계서 - 프론트엔드 동선 재정리 요청

작성일: 2026-06-15
대상: Manus
목적: 부산대 과팅앱의 현재 기능, 화면 흐름, 문제점, 다음 디자인 요청사항을 한 번에 전달한다.

---

## 1. 앱 한 줄 설명

이 앱은 부산대 학생들이 친구와 2명 또는 3명 그룹을 만들고, 상대 그룹을 자동으로 매칭받은 뒤, 만남 전까지 서로의 프로필을 직접 공개하지 않고 하루에 한 장씩 가벼운 정보 카드를 열어보는 그룹미팅 앱이다.

핵심 감정은 단순 소개팅 앱이 아니라, "누가 나올지 모르지만 하루마다 조금씩 단서가 열리는 과팅 대기감"이다.

## 2. 현재 제품 방향

| 항목 | 현재 결정 |
| --- | --- |
| 초기 정책 | 사용자 확보가 우선이라 초기 배포는 완전 무료 베타 |
| 결제/환불 | 현재 사용자 화면에서는 보증금/환불 UX를 뒤로 뺌 |

| 매칭 비중 | 외모, 성격, 키, 체형 4개만 유지 |

## 4. 현재 사용자가 지나가야 하는 앱 흐름

### A. 가입/온보딩

1. 이메일 인증으로 로그인한다.
2. 기본 정보를 입력한다.
3. 이상형 월드컵을 한다.
4. 내 성향 질문을 답한다.
5. 사진을 업로드한다.
6. 상대 성향 선호, 가능 시간, 매칭 비중을 입력한다.
7. 프로필 완성 후 홈으로 들어간다.

현재 문제:

- 로컬 preview에서는 막힘 없이 들어가야 하는데, 일부 화면은 로그인/인증 흐름과 섞여 보이면 검토가 어렵다.
- 온보딩 단계가 서비스 가치와 연결되어 보이지 않고, 사용자가 "왜 이걸 해야 하지?"라고 느낄 수 있다.

### B. 홈/매칭 진입

1. 홈에서 친구 추가 또는 매칭 찾기를 누른다.
2. 친구를 초대해 그룹을 만든다.
3. 각 그룹원이 자기 설정을 완료한다.
4. 그룹 준비가 완료되면 매칭 큐에 들어간다.
5. 큐 화면에서는 남자/여자 그룹이 얼마나 대기 중인지 점 형태로 보여준다.

현재 원하는 홈 방향:

- 홈 첫 화면에 `친구 추가`, `매칭 찾기`, `진행중인 매칭 결과 확인`이 분명히 보여야 한다.
- 진행 중인 매칭이 있으면 사용자가 다시 찾지 않아도 바로 이어갈 수 있어야 한다.

### C. 매칭 확정 전

1. 매칭 후보 또는 상대 그룹이 잡힌다.
2. 내 그룹원이 각자 데일리카드 내용을 작성한다.
3. 그룹 전체 카드 작성 상태를 확인한다.
4. 무료 베타 참여 확인을 한다.
5. 리더가 최종 확인하면 매칭이 확정된다.

현재 구현된 카드 작성 방향:

- 기존의 큰 textarea 하나는 줄이고, 항목별 카드 작성 UI로 바꾸는 중이다.
- 현재 항목 예시는 다음과 같다.
  - 좋아하는 노래
  - 음식 취향
  - 주말/시간 취향
  - 대화 스타일
  - 만남 전 질문
  - 인터넷 논쟁 카드
  - 요즘 빠진 것
- 완료율은 `내 카드 항목 4/7 완료`처럼 항목 기준으로 보여주는 방향이다.
- 그룹 전체는 `3명 중 2명 카드 작성 완료`처럼 별도로 보여줘야 한다.

### D. 매칭 확정 후

1. 약속 시간과 장소를 확인한다.
2. 네이버지도 링크로 장소를 볼 수 있어야 한다.
3. 16:00-20:00 사이 하루 카드 한 장을 직접 뽑는다.
4. 뽑은 카드에서 상대 그룹의 정보를 조금씩 확인한다.
5. 필요하면 매칭 채팅방으로 들어간다.
6. 만남 이후에는 후기/연락처 공개/노쇼 처리 흐름이 이어진다.

현재 문제:

- 데일리카드가 서비스의 핵심 재미인데, 화면상 매력이 부족하다.
- 장소/맛집/네이버지도와 카드가 아직 감각적으로 연결되어 있지 않다.
- 채팅, 카드, 약속 정보가 어디에서 이어지는지 사용자가 한눈에 알기 어렵다.

## 5. 지금 가장 큰 프론트엔드 문제

Manus에게 가장 먼저 봐달라고 할 문제는 코드 품질보다 사용자 동선이다.

1. 첫 화면에서 "이 앱을 왜 써야 하는지"가 직관적으로 안 보인다.
2. 홈에서 다음 행동이 명확하지 않다.
3. 친구 추가, 그룹 생성, 매칭 찾기, 매칭 결과 확인이 기능상 연결되어 있지만 화면상으로는 흩어져 보인다.
4. 데일리카드가 핵심 기능인데 카드 뽑기 게임처럼 설레지 않는다.
5. 음식/논쟁 카드가 텍스트 위주라 재미가 약하다.
6. 일부 화면의 한국어 문구가 깨져 보이는 문제가 있어 신뢰도가 떨어진다.
7. dev preview와 실제 서비스 흐름이 섞여 있어, 디자인 검토자가 어디를 봐야 하는지 헷갈린다.

## 6. 음식 취향 카드 변경 아이디어

현재 음식 취향은 단순 텍스트 입력에 가깝다.

사용자 요청 방향은 다음과 같다.

### 바꾸고 싶은 방향

`음식 취향`을 "부산대 맛집 3곳 고르기" 방식으로 바꾼다.

예상 UX:

1. 사용자가 부산대 근처 맛집 카드 목록을 본다.
2. 가고 싶은 곳 또는 추천하고 싶은 곳 3개를 고른다.
3. 각 맛집 카드에는 이름, 음식 종류, 분위기, 거리, 네이버지도 링크가 있다.
4. 매칭 후 카드가 공개될 때 "상대 그룹이 고른 부산대 맛집" 또는 "우리 둘 다 관심 있는 맛집"처럼 보여준다.
5. 약속 장소 추천/확정과 자연스럽게 연결한다.

### 네이버지도 연동 방식 후보

| 방식 | 설명 | 추천도 |
| --- | --- | --- |
| 1차 MVP: 수동 맛집 리스트 + 네이버지도 링크 | 부산대 맛집 30-50개를 직접 seed로 만들고 각 장소에 Naver Map URL을 붙인다 | 추천 |
| 2차: 네이버 지도 API 검색 | 사용자가 검색해서 장소를 고른다 | 나중 |
| 3차: 장소 추천 자동화 | 매칭 시간/취향/거리 기반으로 장소를 추천한다 | 나중 |

지금은 API 연동보다 수동 맛집 리스트가 더 안전하다. 이유는 빠르게 디자인 검토가 가능하고, 네이버 API 키/정책/검색 품질 문제에 덜 막히기 때문이다.

## 7. 인터넷 논쟁 카드 변경 아이디어

현재 인터넷 논쟁카드는 A/B 텍스트 선택으로 구현되어 있다.

최근 정리된 방향:

- 공개되는 카드는 하루에 한 장일 수 있다.
- 하지만 작성 단계에서는 여러 논쟁 질문에 미리 답해둬야 한다.
- 현재 작성 UI에는 4개 질문이 들어가 있다.
  - 탕수육: 부먹 vs 찍먹
  - 민트초코: 가능 vs 반민초
  - 중식: 짜장 vs 짬뽕
  - 냉면: 물냉 vs 비냉

### 바꾸고 싶은 방향

텍스트 버튼이 아니라 이미지형 선택 카드로 바꾼다.

예시:

| 질문 | 왼쪽 카드 | 오른쪽 카드 |
| --- | --- | --- |
| 탕수육 | 찍먹 이미지 | 부먹 이미지 |
| 중식 | 짜장면 이미지 | 짬뽕 이미지 |
| 민초 | 민트초코 디저트 이미지 | 반민초/초코 디저트 이미지 |
| 냉면 | 물냉면 이미지 | 비빔냉면 이미지 |

이미지는 실제 사람 얼굴/프로필 사진을 쓰지 않는다. 음식 이미지는 생성 이미지 또는 저작권 문제가 없는 소스만 사용한다.

원하는 느낌:

- 싸구려 퀴즈앱 느낌이 아니라, "카드를 고르는 맛"이 있어야 한다.
- 선택하면 카드가 살짝 들어가거나 빛나는 식으로 반응이 있어야 한다.
- 결과 공개 때는 우리 그룹과 상대 그룹의 일치/불일치가 귀엽게 보여야 한다.

## 8. Manus에게 요청할 결과물

Manus에게 바로 코드 구현을 맡기기보다, 먼저 프론트엔드 동선과 화면 설계를 추천받는 것이 목표다.

요청할 결과물:

1. 전체 사용자 여정 지도
   - 신규 사용자
   - 친구 초대 사용자
   - 매칭 대기 사용자
   - 매칭 확정 후 사용자
2. 홈 화면 재구성안
   - 친구 추가
   - 매칭 찾기
   - 진행중인 매칭
   - 오늘 뽑을 카드
3. 그룹 생성/큐 진입 화면 재구성안
   - 초대 상태
   - 멤버별 준비 상태
   - 큐 진입 조건
   - 큐 진입 완료 화면
4. 매칭 상세 화면 재구성안
   - 카드 작성
   - 무료 베타 참여 확인
   - 매칭 확정
   - 약속 장소/시간
   - 하루 1장 카드 뽑기
   - 채팅 진입
5. 데일리카드 작성 화면 재구성안
   - 텍스트 입력을 줄이고 항목별 카드/선택 UI 강화
   - 음식 취향은 부산대 맛집 3곳 선택으로 변경
   - 인터넷 논쟁은 이미지형 A/B 카드로 변경
6. 화면별 우선순위
   - 지금 당장 고쳐야 하는 것
   - 나중에 해도 되는 것
   - 버려도 되는 것

## 9. Manus가 지켜야 할 제약

1. 초기 배포는 무료 베타다. 결제/환불을 메인 동선에 다시 올리지 않는다.
2. 프로필 비공개 자동확정 과팅 앱이라는 컨셉을 유지한다.
3. 매칭 전에는 상대 얼굴/개인 신상을 직접 보여주는 방향으로 가지 않는다.
4. 음식/논쟁/데일리카드는 가벼운 흥미 요소이지, 사용자를 피곤하게 만드는 설문지가 되면 안 된다.
5. 로컬 디자인 검토에서는 `/dev/preview`에서 막힘 없이 전체 화면을 확인할 수 있어야 한다.
6. 네이버지도 연동은 최종적으로 필요하지만, 1차 디자인안에서는 수동 URL 기반으로 시작해도 된다.
7. DB/API 대수술은 지금 요청 범위가 아니다. 먼저 UX 흐름과 화면 구조부터 잡는다.

## 15. Manus에게 꼭 알려야 할 세부 꼬임

1. `/dev/preview`는 로컬 디자인 검토용 허브지만, 일부 화면은 각자 다른 방식으로 dev mock을 보여준다.
2. `/dev/preview`의 설명에는 채팅이 보류처럼 남아 있지만, 현재 채팅 화면/API/DB 마이그레이션은 이미 기본 구현이 들어가 있다.
3. 프로필 완료와 매칭 준비가 분리되어 있다.
   - 기본 온보딩: 학교/기본정보/월드컵/성향/사진
   - 매칭 준비: 상대 성향 선호/가능 시간/매칭 비중
4. 데일리카드는 용어가 두 겹이다.
   - 매칭 확정 전: 사용자가 미리 작성하는 `내 카드 항목`
   - 매칭 확정 후: 16:00-20:00에 직접 뽑는 `하루 1장 공개 카드`
5. 네이버지도 SDK 연동은 아직 없다. 현재는 `venue_map_url` 또는 `venues.map_url`을 외부 링크로 여는 수준이다.
6. 보증금 관련 API/테이블명은 남아 있지만, 사용자 화면 정책은 무료 베타 참여 확인으로 바뀌었다. Manus는 결제/환불을 메인 동선에 다시 올리면 안 된다.
7. 일부 문서/화면에서 한국어가 깨져 보이는 문제가 있다. 프론트 신뢰도에 직접 영향을 주므로 디자인 정리와 별도로 복구가 필요하다.

## 16. 맛집/지도 기능의 실제 구현 기준

현재 DB 쪽에는 `venues`, `match_meetings`, `venues.map_url`을 쓸 수 있는 기반이 있다. 다만 부산대 맛집 seed 데이터는 현재 코드 검색상 확정된 insert가 보이지 않는다.

따라서 Manus에게는 다음 기준으로 설명한다.

### 1차 MVP

- 부산대 맛집 30-50개를 운영자가 직접 선별한다.
- 각 장소는 `venues`에 저장한다.
- `map_url`에는 네이버지도 장소 링크를 수동으로 넣는다.
- 사용자는 음식 취향 카드에서 맛집 3곳을 고른다.
- 매칭 후에는 "상대 그룹이 고른 맛집", "우리와 겹치는 맛집", "약속 장소 후보"처럼 보여준다.

### 나중 확장

- 네이버 지역 검색 API로 장소 검색을 붙인다.
- Naver Maps JS SDK로 지도 미리보기와 거리 표시를 붙인다.
- 매칭 시간, 취향, 거리, 영업시간 기반으로 장소를 추천한다.

지금 당장 추천은 1차 MVP다. API 연동부터 하면 키, 한도, 과금, 검색 품질, 지도 UI까지 한 번에 얽혀서 프론트 동선 정리가 늦어진다.
