# 성준 미팅 논의 안건 - 2026-06-01

## 회의 목적

현재 로컬에서는 카카오 로그인 이후 실제 연동이 막혀 있으므로, 오늘은 **기능 방향과 우선순위 결정**에 집중한다. 화면은 로컬 URL로 직접 확인하고, 백엔드/결제/매칭 연동 전에도 확정할 수 있는 UX와 정책을 먼저 정한다.

## 1. 로그인과 학교 인증

### 결정할 것

- 카카오 로그인만 먼저 갈지, Google/GitHub 등 소셜 로그인도 열지.
- 카카오 로그인 직후 바로 학교 이메일 인증을 강제할지.
- `pusan.ac.kr`만 허용할지, 부산대 관련 다른 도메인도 받을지.
- 인증 전 사용자가 볼 수 있는 범위: 랜딩만, 프로필 입력까지, 또는 일부 미리보기까지.

### 추천

MVP는 **카카오 로그인 + 부산대 학교 이메일 인증 필수**로 간다. 인증 전에는 프로필/그룹/매칭 진입을 막고, 로컬 개발용 프리뷰는 별도 `/debug/*` 라우트로 분리한다.

## 2. 온보딩 순서

현재 흐름:

1. 기본 정보
2. 이상형 월드컵
3. 사진 등록
4. 성격 설문
5. 가능 시간
6. 매칭 선호도
7. 완료

### 결정할 것

- 이 순서 그대로 갈지.
- 사진 등록을 필수로 할지, 나중에 하기 허용할지.
- 이상형 월드컵을 필수로 할지, 기본 선호도만으로 시작 가능하게 할지.
- 온보딩 중 이탈 시 어디까지 저장할지.

### 추천

매칭 품질을 위해 기본 정보, 성별, 학교 인증, 이상형 월드컵, 사진은 필수로 둔다. 성격/시간/선호도는 최소 입력 후 나중에 수정 가능하게 둔다.

## 3. 이상형 월드컵 남자/여자 분리

### 현재 이슈

로컬 화면은 `/profile/worldcup` 하나만 있고, 개발 기본값 때문에 한쪽 후보만 바로 보인다. 카카오/프로필 연동 전에는 남자 후보와 여자 후보를 각각 직접 띄워 비교하기 어렵다.

### 결정할 것

- 남자 후보/여자 후보 각각 몇 장으로 시작할지: 16강, 32강, 64강.
- 남성 사용자는 여성 후보를 보고, 여성 사용자는 남성 후보를 보는 현재 방식으로 확정할지.
- 동성 매칭/확장 가능성은 지금 닫을지.
- 후보 이미지는 실사풍/일러스트풍/AI 생성 이미지 중 무엇으로 갈지.
- 월드컵 결과를 사용자가 볼 수 있게 할지, 내부 매칭 벡터로만 쓸지.

### 추천

MVP는 성별 기반 반대 성별 후보만 노출한다. 다만 회의/QA용으로는 `/debug/worldcup?target=male`, `/debug/worldcup?target=female` 같은 개발용 프리뷰를 추가해서 남자/여자 후보를 따로 확인할 수 있게 한다.

## 4. 사진 등록과 외모 분석

### 결정할 것

- 사진 최소 개수: 1장, 2장, 3장.
- AI 외모 점수를 사용자에게 보여줄지, 내부 매칭용으로만 쓸지.
- 사진 검수 기준: 얼굴 없음, 여러 명 사진, 부적절 이미지 차단 방식.
- 외모 분석 실패 시 가입을 막을지, 재시도 안내만 할지.

### 추천

사용자에게 외모 점수 숫자는 보여주지 않는다. “매칭 품질을 위해 사진 특징을 분석한다” 정도로만 설명하고, 내부 점수/벡터는 비노출한다.

## 5. 그룹 매칭 방식

### 결정할 것

- 그룹 크기: 2명 고정, 3명 고정, 2~3명 허용.
- 리더만 매칭 큐에 넣을 수 있게 할지.
- 친구 초대 방식: 전화번호, 앱 친구, 링크 초대 중 MVP 포함 범위.
- 그룹 성별은 리더 기준으로 자동 결정할지, 멤버 전체 성별 검증을 할지.

### 추천

MVP는 2~3명 허용, 리더만 큐 진입 가능, 링크 초대와 전화번호 초대를 우선 지원한다. 앱 친구 시스템은 있으면 좋지만 MVP 핵심은 아니다.

## 6. 보증금과 결제

### 결정할 것

- 보증금 금액: 현재 기준 20,000원 유지 여부.
- 결제 완료 전 매칭 큐 진입 차단 여부.
- 토스페이먼츠 연동 시점.
- 결제 실패/취소/중복 결제 처리 정책.

### 추천

보증금은 매칭 큐 진입 전 필수로 둔다. 실제 결제 연동 전까지는 mock 결제로 UX를 확정하고, 이후 Toss sandbox로 바꾼다.

## 7. 구걸 환불 UX

### 현재 컨셉

만남 종료 후 사용자가 보증금을 환불받을 금액을 직접 선택한다. 전액 환불을 누르면 캐릭터가 3,000원, 2,000원, 1,000원이라도 남겨달라고 단계적으로 부탁하는 UX가 있다.

### 결정할 것

- 이 구걸 UX를 실제 서비스에 넣을지.
- 톤을 코믹하게 유지할지, 너무 가벼워 보이면 줄일지.
- 전액 환불을 몇 번까지 막거나 설득할지.
- 0원 환불 선택 시 사유 입력을 필수로 할지.
- 상대 그룹에게 “상대가 0원 환불했다” 같은 알림을 보낼지.

### 추천

컨셉은 유지하되 문구는 과하지 않게 다듬는다. 돈과 신뢰가 걸린 화면이라 장난이 너무 세면 반감이 생길 수 있다. “운영비 후원”에 가깝게 표현하고, 최종 전액 환불은 언제든 가능해야 한다.

## 8. 매칭 엔진과 결과 생성

### 결정할 것

- 매칭 실행 시간: 일요일 14:00 유지 여부.
- 매칭 확정 후 시간/장소가 바로 정해지는지.
- `match_meetings.scheduled_start`가 언제 생성되는지.
- 그룹 양쪽 리더가 모두 확정해야 최종 확정인지.
- 매칭 실패 그룹은 다음 회차로 자동 이월할지.

### 성준님에게 꼭 물어볼 것

1. `match_meetings.scheduled_start`는 매칭 confirm 시점에 이미 확정돼 있나?
2. 매칭 큐 진입 조건 검사는 `enter_match_pool` RPC 안에서 서버 강제로 막는 게 맞나?
3. 매칭 confirmed 트리거에 데일리 카드/협동 활동 배정을 붙여도 되나?

## 9. 데일리 카드 기능

### 기능 요약

매칭 확정 후 만남 전까지 5~7일 동안 상대 그룹에 대한 취향 카드, 힌트, 협동 활동을 매일 조금씩 공개하는 기능.

### 결정할 것

- MVP에 포함할지, 2차 기능으로 미룰지.
- 공개할 정보 범위: 취향, 성격, 장소 힌트, 활동 미션.
- 프로필 비공개 원칙과 충돌하지 않는지.
- 매일 09:00 공개 같은 고정 스케줄을 둘지.

### 추천

MVP 핵심은 매칭 성사이므로 데일리 카드는 1차 출시 이후로 미루는 편이 안전하다. 다만 DB/매칭 엔진 경계는 지금 성준님과 합의해 둬야 나중에 덜 뜯는다.

## 10. 노쇼와 출석 체크

### 결정할 것

- GPS 체크인 허용 반경.
- 체크인 가능 시간: 약속 30분 전부터 2시간 후까지 등.
- 한 명만 노쇼여도 그룹 전체 보증금 처리에 영향을 줄지.
- 노쇼 보증금을 출석자에게 분배할지, 운영비로 둘지.

### 추천

MVP는 단순하게 간다. 약속 장소 반경 내 체크인만 출석으로 인정하고, 노쇼 보증금은 출석자에게 균등 보상하는 정책을 먼저 테스트한다.

## 11. 관리자/운영

### 결정할 것

- 운영자가 수동으로 수정할 수 있어야 하는 항목:
  - 외모 분석 점수
  - 부적절 사진
  - 신고/차단
  - 매칭 취소
  - 보증금 환불/몰수
- 출시 전 관리자 페이지를 만들지, Supabase 콘솔로 임시 운영할지.

### 추천

초기에는 Supabase 콘솔 + 최소 관리자 SQL/RPC로 운영하고, 반복 작업이 보이면 관리자 페이지로 빼는 게 빠르다.
