# 팀장방 운영 절차

최신 기준: 2026-06-15

이 문서는 팀장방이 하위 직원을 어떻게 쓰고, 언제 직접 하고, 언제 멈춰서 보고할지 정한 실행 절차다.

## 1. 작업을 받으면 먼저 분류한다

작업을 받으면 바로 코드를 고치기 전에 아래 네 가지로 나눈다.

| 분류 | 예시 | 담당 |
| --- | --- | --- |
| 구조/흐름 문제 | 홈에서 매칭으로 가는 길이 이상함, preview가 로그인으로 튕김 | Noether |
| UX/디자인 문제 | 화면이 허접함, 문구가 이상함, 카드 선택이 직관적이지 않음 | McClintock |
| DB/API 문제 | Supabase migration, RPC, RLS, API route 위험 | Bernoulli |
| 구현 문제 | 확정된 UI를 실제 컴포넌트로 만들기 | Darwin |
| 이미지/에셋 문제 | 논쟁카드 이미지, 음식카드 이미지, public 에셋 정리 | Pascal |

## 2. 팀장방이 직접 해야 하는 일

아래는 하위 직원에게 넘기지 않는다.

- 최종 방향 결정
- 여러 worker 결과의 충돌 판단
- 같은 파일을 누가 맡을지 정하는 일
- 커밋에 넣을 파일과 제외할 파일 분류
- production Supabase 적용 여부 판단
- 사용자에게 최종 보고

## 3. 하위 직원에게 맡기는 기준

하위 직원에게 맡겨도 되는 작업:

- 읽기 전용 조사
- 서로 다른 파일을 수정하는 구현
- 이미지 생성처럼 코드와 분리된 작업
- DB/API 위험 검토처럼 전문 영역이 분명한 작업

하위 직원에게 맡기면 안 되는 작업:

- 바로 다음 단계가 그 결과에 막혀 있는 급한 판단
- 같은 파일을 여러 명이 동시에 수정해야 하는 작업
- 제품 방향 자체를 확정하는 작업
- 사용자가 방금 화낸 문제의 핵심 원인을 팀장방이 아직 이해하지 못한 상태

## 4. 작업 시작 전 사용자 보고

작업 시작 전에 아래 형식으로 짧게 보고한다.

```text
이번 작업 배정:
- Noether: 읽기 전용, 앱 흐름 조사
- McClintock: 읽기 전용, UX 문제 정리
- Darwin: app/match/[id]/page.tsx만 수정
- Pascal: public/daily-cards/debate/만 수정
- 팀장방: 취합/검증/커밋 판단
```

사용자가 “하위 직원 써라”, “worker에게 맡겨라”, “팀원 나눠서 해라”라고 명령한 경우에만 하위 직원을 부른다.

## 5. 하위 직원 시작 프롬프트 구성

하위 직원에게는 항상 아래를 포함한다.

```text
역할명:
목표:
반드시 읽을 문서:
- docs/coordination/AGENT_ROLES.md
- docs/coordination/TEAM_CONTEXT.md
- docs/coordination/WORKER_REPORT_TEMPLATE.md

수정 허용 범위:
수정 금지 범위:
검증 방법:
보고 형식:
주의: 커밋하지 말 것. 기존 사용자 변경을 되돌리지 말 것.
```

역할별 자세한 프롬프트는 `ROLE_PROMPTS.md`를 사용한다.

## 6. worker 진행 중 팀장방 행동

worker가 도는 동안 팀장방은 겹치지 않는 작업을 한다.

예시:

- worker가 UX 감사를 하는 동안 팀장방은 `git status`와 변경 묶음 확인
- worker가 이미지 생성하는 동안 팀장방은 UI 연결 파일 조사
- worker가 DB 검토하는 동안 팀장방은 프론트 영향 범위 정리

반대로, worker 결과를 기다려야만 다음 행동을 할 수 있으면 직접 처리하거나 기다린다.

## 7. 완료 후 취합

worker가 끝나면 팀장방이 아래 순서로 취합한다.

1. 보고서가 역할 범위 안에 있는지 확인
2. 수정 파일이 허용 범위 안인지 확인
3. 같은 파일 충돌 여부 확인
4. 검증 명령과 route 확인 결과 확인
5. 부족하면 팀장방이 보완하거나 같은 worker에게 재지시
6. 사용자에게 완료 보고

## 8. 커밋 기준

커밋 전에는 항상 보고한다.

```text
커밋에 넣을 파일:
- ...

이번 커밋에서 제외할 파일:
- ...

staged diff 요약:
- ...

추천 커밋 메시지:
- ...
```

사용자가 커밋하라고 명령하기 전까지 커밋하지 않는다.

## 9. 검증 기준

프론트 구현 후 기본 검증:

```bash
npm run typecheck
npm run lint
```

매칭/데일리카드 관련이면 추가:

```bash
npm run test:matching
```

route 확인:

```text
http://127.0.0.1:3003/dev/preview
http://127.0.0.1:3003/match/dev-match-pending
http://127.0.0.1:3003/match/dev-match-1
```

DB 작업은 production이 아니라 local 또는 disposable staging에서만 검증한다.

## 10. 막혔을 때 처리

막히면 같은 지시를 반복하지 않는다.

```text
막힌 지점:
원인 후보:
시도한 것:
더 필요한 정보:
팀장방 추천:
```

사용자가 결정해야 하는 내용은 쉬운 말로 따로 보고한다.
