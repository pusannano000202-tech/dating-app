# 가매칭 카드-보증금 흐름 설계

작성일: 2026-06-02

## 목적

기존의 “보증금 결제 후 매칭 큐 진입” 흐름을 “가매칭 후 카드 작성, 그 다음 보증금 결제” 흐름으로 바꾼다. 사용자는 상대가 잡혔다는 기대감이 생긴 뒤 자기 카드를 작성하고 보증금을 넣는다. 실제 만남 전 일주일은 하루 하나씩 상대 정보를 알아가는 구간으로 만든다.

## 새 전체 흐름

```text
그룹 생성
  -> 매칭 큐 진입
  -> 시스템 가매칭
  -> 카드 작성 요청
  -> 카드 작성 완료
  -> 보증금 결제
  -> 양쪽 조건 충족 시 매칭 확정
  -> D-6 ~ D-0 데일리 카드 공개
  -> 실제 만남
```

## 1:1 과팅

1:1에서는 상대가 한 명이므로 카드 공개 대상도 한 명이다.

- 익명 별칭은 선택 사항이지만, MVP에서는 일관성을 위해 별칭을 부여한다.
- 예: `오소리`라는 별칭으로 상대 카드가 열린다.
- 공개 정보는 취향, 성격, 이상형, 밸런스 게임 답변처럼 신원 특정이 어려운 정보만 포함한다.
- 실명, 사진, 전화번호는 기존 비공개 원칙을 유지한다.

## 그룹 과팅

3:3 같은 그룹 과팅에서는 “상대 그룹 평균 정보”만 보여주지 않는다. 상대 멤버 각각에게 가상 닉네임을 부여하고, 한 명 한 명에 대한 카드가 열린다.

예시:

| 실제 상대 | 우리 그룹에 보이는 별칭 |
|---|---|
| 상대 멤버 A | 오소리 |
| 상대 멤버 B | 꿀벌 |
| 상대 멤버 C | 개구리 |

우리 팀 3명은 매일 `오소리 카드`, `꿀벌 카드`, `개구리 카드`를 본다. 같은 매칭 안에서는 모든 우리 팀원이 같은 별칭 매핑을 본다. 반대쪽 그룹도 우리 팀 멤버들에게 별칭을 부여해서 같은 방식으로 본다.

## 카드 작성 시점

카드는 매칭 큐 진입 전에 강제하지 않는다. 가매칭이 잡힌 뒤 작성한다.

이유:

- 상대가 생긴 뒤 작성해야 사용자가 더 성의 있게 쓴다.
- 매칭이 안 잡힌 상태에서 긴 입력을 강제하면 이탈 가능성이 높다.
- 보증금 결제 전 카드 작성은 결제 전환 설득 장치가 된다.

## 보증금 시점 변경

기존:

```text
보증금 결제 -> 매칭 큐 진입
```

변경:

```text
매칭 큐 진입 -> 가매칭 -> 카드 작성 -> 보증금 결제 -> 확정
```

서버 규칙:

- `enter_match_pool`에서는 보증금 결제를 요구하지 않는다.
- 가매칭 후 확정 단계에서 활성 멤버 전원의 카드 작성과 보증금 결제를 검사한다.
- 양쪽 그룹 모두 조건을 만족해야 `confirmed` 상태가 된다.

## 상태 모델

현재 `matches.status`는 `pending`, `confirmed`, `completed`, `cancelled`, `no_show` 중심이다. 새 흐름에서는 다음 상태를 추가한다.

| 상태 | 의미 |
|---|---|
| `card_pending` | 가매칭은 되었고, 카드 작성/보증금 결제를 기다리는 상태 |
| `pending` | 카드와 보증금 조건은 충족했고, 양쪽 리더 확정을 기다리는 상태 |
| `confirmed` | 양쪽 리더 확정 완료, 만남 확정 상태 |

MVP에서는 기존 `pending`을 당장 유지하면서 UI에서 카드/보증금 대기 상태를 표현할 수 있다. 다만 DB와 운영 관점에서는 `card_pending`을 추가하는 편이 명확하다.

## 새 데이터

### `match_member_aliases`

매칭 안에서 상대 멤버를 익명 별칭으로 보여주기 위한 테이블.

필드 초안:

- `match_id`
- `viewer_group_id`
- `target_user_id`
- `alias`
- `alias_theme`
- `sort_order`

같은 `match_id + viewer_group_id + target_user_id`는 하나만 존재한다.

### `user_daily_card_content`

각 사용자가 작성하는 카드 원본.

예:

- 좋아하는 음식
- 좋아하는 노래
- 주말 취향
- MBTI와 한 줄 소개
- 이상형 관련 공개 가능한 요약
- 밸런스 게임 답변

### `match_daily_card_schedule`

매칭마다 어떤 날짜에 어떤 카드가 열릴지 저장한다.

### `match_coop_activity`

그룹끼리 같이 하는 밸런스 게임, 공동 활동 등을 저장한다.

## 공개 규칙

- 카드에는 개인 신원 특정 정보가 들어가면 안 된다.
- 전화번호, SNS ID, 학번, 구체적인 소속, 정확한 사진은 공개하지 않는다.
- 3:3에서는 개인별 별칭 단위로 공개하지만, 실명과 연결하지 않는다.
- D-0 또는 만남 직전에도 실명/연락처 공개는 기존 연락처 공개 정책을 따른다.

## UI 변경

### 그룹 화면

기존 보증금 선결제 블록을 제거하거나 뒤로 미룬다.

새 CTA:

- “매칭 큐에 들어가기”
- 조건: 리더, 최소 2명, 그룹 상태 `forming`
- 보증금 미결제여도 가능

### 매칭 상세 화면

가매칭 상태일 때 보여줄 것:

- “상대 그룹이 잡혔어요”
- “카드를 작성하면 보증금 결제로 넘어갈 수 있어요”
- 내 카드 작성 상태
- 우리 그룹 멤버 카드 작성 상태
- 보증금 결제 상태
- 모든 조건 충족 시 리더 확정 버튼

### 카드 작성 화면

최소 MVP:

- 짧은 선택형/텍스트형 카드 5개
- 저장 후 수정 가능
- 그룹 과팅에서도 멤버별 원본 카드는 각자 작성

## 1차 구현 범위

이번 1차에서는 전체 데일리 공개까지 완성하지 않고, 흐름의 핵심 경계를 먼저 바꾼다.

1. 계획서 정리 폴더 생성
2. `enter_match_pool`의 보증금 요구 제거
3. 그룹 화면에서 보증금 선결제 CTA를 큐 진입 뒤/가매칭 뒤로 이동
4. 가상 별칭 테이블/헬퍼 설계 추가
5. 카드 작성/보증금 확정 구현 계획 작성

## 나중 구현

- `assign_daily_cards(match_id)` RPC
- `reveal_due_daily_cards` cron
- 카드 작성 API/UI
- 매칭 상세의 카드 캘린더 UI
- 실시간 밸런스 게임
- 관리자 카드 문항 관리
