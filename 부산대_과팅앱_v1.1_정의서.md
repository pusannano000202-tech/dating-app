## 4. 매칭 시스템 상세

### 4.1 외모 점수 (2단 구조)

**1단 — 절대점수 (AI 자동 산출)**

- 모델: **SCUT-FBP5500 + ResNet** (GitHub 사전학습 모델)
- 사용자 사진 3장 평균 → 0~100점 산출
- 동시에 외모 타입 분류: **귀여움 / 청순 / 시크 / 훈훈** (CLIP + 한국 연예인 참조 임베딩 하이브리드)

**2단 — 상대점수 (취향 가중치)**

- 사용자 A의 취향이 "귀여움"이면, B가 "귀여움" 타입일 때 부스트
- 일치 시 ×1.3, 불일치 시 ×0.8 (시스템 고정 배율)

### 4.2 성격 점수

- **MBTI식 고정 설문지** (IPIP-NEO 50문항 한국어판 권장)
- 빅5 5차원 (외향성/개방성/성실성/친화성/신경성) 점수화
- AI 인터뷰는 폐기 (비용/구현 복잡도)

### 4.3 가중치 시스템 (2단계 분배)

**1단계 — 항목 가중치 (15점 분배)**

- 항목: 외모 / 성격 / 몸매 / 키 / 학과매력 / 취미일치 등 (4~6개)
- 5점 균등 = 가중치 1.0 (의미 없음)
- 외모 8 / 성격 2 / 몸매 5 → 외모 1.6, 성격 0.4, 몸매 1.0

**2단계 — 외모 내 절대 vs 상대 분배 (10점 분배)**

- 5/5 = 가중치 1.0/1.0
- 상대 7 / 절대 3 → 1.4 / 0.6
- 극단값(10/0, 0/10) 허용하되 8점 이상 몰빵 시 경고 1회 띄움 *(미확정)*

### 4.4 매칭 알고리즘 (1:1)

**핵심**: 양방향 점수 계산 + 헝가리안 알고리즘

**Step 1 — 양방향 점수 계산**

- 사용자 A → B 점수: A의 가중치 적용해서 B 평가
- 사용자 B → A 점수: B의 가중치 적용해서 A 평가
- 100명 × 100명 → 10,000개 점수 (양방향이라 사실상 20,000개)

**Step 2 — 쌍방 점수 결합**

- **조화평균** 사용: `2 × (A→B) × (B→A) / ((A→B) + (B→A))`
- 한쪽만 높고 한쪽이 낮으면 점수 확 떨어짐 → 균형 매칭 자동 보장

**Step 3 — 임계값 필터**

- 양쪽 모두 60점 이상인 페어만 후보 *(임계값 미확정)*
- `excluded_pairs`(전 매칭 기록) 제외

**Step 4 — 헝가리안 알고리즘**

- `scipy.optimize.linear_sum_assignment` 사용
- 전체 합이 최대인 매칭 조합 자동 도출
- 100×100 계산 시간 < 1초

### 4.5 매칭 주기

- **주 1회 배치 처리** (예: 일요일 자정)
- 그 주에 매칭 신청한 사람 전원 한 번에 처리
- 사용자 UX: "기대감" 유지 + 신중한 매칭 인식

### 4.6 시간/장소 자동 지정

**시간 결정**:

- 사용자가 가입/매칭 신청 시 가능 시간대 미리 입력 (요일×시간 그리드)
- 매칭 확정 시 양측 가능 시간 교집합에서 시스템이 자동 선택

**장소 결정**:

- 부산대 근처 카페/식당 DB (관리자 큐레이션, 30~50개)
- 시스템이 3개 후보 제시 → 양측 동시 투표 → 다수결 *(미확정)*
- 또는 완전 자동 1개 지정 *(미확정)*

⚠️ 매칭 확정 후 만남 전까지 **앱 내 채팅 불가** (강제 만남 컨셉)

## 5. 안전/페널티 시스템

### 5.1 거짓말 신고 시스템

**구조**:

- 가입 시 키/머리숱/체형 등 자가 입력
- 가입 시 **경고 동의 체크박스 필수** ("정보 거짓 입력 시 즉시 페널티")
- 만남 후 상대가 거짓말 신고 가능
- **1회 신고 = 즉시 절대 외모점수/성격점수 자동 차감**

**위험 관리**:

- 신고당하면 본인에게 알림 (7일 이의제기 기간)
- 운영자(성준+형) 수동 검토 후 페널티 확정/취소
- 무고 방지: 거짓 신고가 누적되는 신고자도 메타 페널티

### 5.2 전 매칭/연인 차단 (excluded_pairs)

**왜 필요**:

- 매칭 점수가 잘 맞아서 만났던 두 사람이 헤어진 후, 다시 매칭되면 큰 사고
- 매칭 알고리즘은 "잘 맞는 점수"로만 판단 → 옛 연인이 다시 매칭될 가능성 ↑

**구현**:

- 한 번이라도 매칭되어 만남이 성사된 두 사람 → `excluded_pairs` 테이블에 영구 기록
- 다음 매칭에서 이 페어는 점수 계산에서 제외 (또는 -∞)
- 사용자가 명시적으로 해제 요청 시만 다시 매칭 가능 *(옵션, 미확정)*

### 5.3 보증금 시스템 (= 노쇼 페널티 + 수익 모델)

기존 데이팅앱의 가장 큰 문제는 **노쇼**. 이를 해결하면서 동시에 앱 운영비/수익을 확보하는 구조.

#### 5.3.1 기본 메커니즘

- **매칭 확정 시 양측 각자 보증금 결제** (금액 *TBD*: 2~3만원 사이 검토 중)
- 만남이 정상 성사되면 → 양측 모두 보증금 환불
- 한쪽만 장소에 나타나고 다른 쪽이 노쇼 → **노쇼한 사람의 보증금이 출석한 사람에게 전액 이전**
- 양쪽 다 노쇼 → 보증금 양측 모두 몰수 (운영사 수익)

#### 5.3.2 출석 검증 (가장 어려운 부분, 사기 방지)

*(검증 방식 미확정 — 후보 4개)*

- (a) **GPS 체크인**: 앱이 약속 시각에 매칭 장소 반경 30m 안에서 사용자 위치 확인
- (b) **QR 코드**: 부산대 근처 제휴 카페에 QR 부착 → 양측이 현장에서 스캔
- (c) **양측 상호 인증**: 양쪽이 앱에서 "상대방 만났음" 동시 버튼 누름 — *단, 담합 사기 위험*
- (d) **위 (a)~(c)의 조합**: 최소 2개 충족 시 출석 인정
- **권장**: (a) + (c) 조합 — GPS는 위치 확인용, 양측 동시 확인은 상호 검증용

#### 5.3.3 수익 모델 (수고비 / 복비 방식)

충현님 아이디어: **점집 복비 형식**

- 만남이 정상 성사된 후, 보증금 환불 시점에 시스템이 물음
  > "이번 매칭에 만족하셨나요? 받으실 환불 금액을 직접 정해주세요. (0원 ~ 3만원)"
  >
- 사용자가 직접 환불 금액 선택 → 나머지는 운영사 수수료로 입금
- 한국식 미신/감사 정서 활용: 마음에 들면 일부 또는 전액 안 돌려받음
- 정말 마음에 안 들면 전액 환불 가능 (사용자 선택권 보장 → 신뢰)

#### 5.3.4 보증금 시스템 외 추가 페널티

- 보증금 외에도 **노쇼 N회 누적** 시 다음 매칭 N주 제한
- 거짓 출석 신고(GPS 조작 등) 적발 시 영구 정지

## 8. 외모 AI 기술 스택 (가장 중요한 부분)

### 8.1 절대 외모점수 산출

- **SCUT-FBP5500 + ResNet50** (GitHub에 사전학습 모델 다수 공개)
- Python + PyTorch로 추론
- 사진 3장 평균 → 0~100점 정규화
- 비용: 무료 (모델만 받아서 본인 서버에서 추론)

### 8.2 외모 타입 분류 (귀여움/청순/시크/훈훈)

- **하이브리드 접근**:
  1. **CLIP zero-shot** (한국어판 KoCLIP 권장): 텍스트 프롬프트 유사도
  2. **참조 연예인 임베딩** (InsightFace): 각 타입별 한국 연예인 5~10명 라벨링 → 코사인 유사도
- 두 점수 평균으로 최종 타입 결정

### 8.3 비용

- 모든 모델 오픈소스, **추가 비용 0원**
- 단, 추론용 서버 (GPU 없어도 가능, CPU 추론 가능) 필요

## 11. 미해결 (다음 결정 필요) 항목

1. **항목 가중치 분배 시 사용 항목 확정**: 외모/성격/몸매/키/학과매력/취미일치 중 몇 개?

4. **시간/장소 지정 — 완전 자동 vs 3개 후보 투표**
5. **그룹 매칭 양방향 알고리즘** (충현 추가 검토 중)

### 보증금 시스템 관련

7. **보증금 금액**: 2만원 vs 3만원 vs 변동제 (장소에 따라)

9. **수고비 환불 UX**: 0원~3만원 슬라이더 vs 정해진 옵션 (전액/반액/0원) vs 완전 자유 입력
10. **결제/환불 처리 수단** (카카오페이/토스/카드)
11. **학교 프로젝트 단계 결제 처리**: 실제 결제 vs 모사 DB 트랜잭션

### 운영 관련

12. **운영비 분담**: 충현/성준 절반? 보증금 시스템 운영 후엔 수익이 들어오니 별 이슈 아닐 수도
